## 摘要
LLM编码代理越来越多地被部署在真实生产基础设施上自主行动。它们执行Shell命令、修改代码仓库、调用操作API。然而，完成任务并不足以保证安全。一个错误行动可能导致严重后果。现有的代理基准测试主要强调任务完成，对于代理在温和但欠指定指令下的行为表现未作探讨。我们提出了UnderSpecBench，一个用于衡量编码代理（如Claude Code、Codex和OpenCode）在DevOps任务中行动边界违规的基准测试。UnderSpecBench包含69个任务族，这些任务族基于记录在案的事件、CVE或工具行为，并横跨四个DevOps能力域和九个操作控制面。为了将欠指定性与任务难度分离，每个任务保持相同的环境和真实安全行动，同时沿三个维度变化指令：意图清晰度、目标确定性以及爆炸半径。由此产生的2208个提示变体使用确定性、基于副作用的预言机进行评估，区分安全成功、错误目标和过度范围结果；非行动运行进一步分类为澄清、拒绝或延迟。在五种使用OpenCode、Claude Code和Codex的代理×模型配置中，评估结果显示欠指定性主要不会导致代理失败，而是使它们进行猜测。55.8-67.8%的运行违反了至少一个边界。目标欠指定性显著降低了行动质量，而爆炸半径提示几乎未减少行动倾向。这些发现表明，以完成为中心的评估可能夸大了安全自主性，并促使在模型、框架和系统层面进行缓解。

## Abstract
LLM coding agents are increasingly deployed to act autonomously on real production infrastructure. They execute shell commands, modify repositories, and call operational APIs. However, completing a task is not sufficient for safety. A wrong action can cause severe consequences. Existing agent benchmarks largely emphasize task completion, leaving open how agents behave under benign but underspecified instructions.   We present UnderSpecBench, a benchmark for measuring action-boundary violations in coding agents (i.e., Claude Code, Codex, and OpenCode) on DevOps tasks. UnderSpecBench includes 69 task families grounded in documented incidents, CVEs, or tool behavior and organized across four DevOps capability domains and nine operational control surfaces. To isolate underspecification from task difficulty, each task keeps the same environment and ground-truth safe action while varying the instruction along three axes: intent clarity, target certainty, and blast radius. The resulting 2,208 prompt variants are evaluated with deterministic, side-effect-based oracles that separate Safe Success, Wrong Target, and OverScope outcomes; non-action runs are further classified as clarification, refusal, or deferment.   Across five agent x model configurations using OpenCode, Claude Code, and Codex, the evaluation results show that underspecification does not mainly make agents fail; it makes them guess. 55.8-67.8% of runs violate at least one boundary. Target underspecification sharply degrades action quality, while blast-radius cues barely reduce action propensity. These findings show that completion-centric evaluation can overstate safe autonomy and motivate mitigations at the model, harness, and system layer.

- **研究动机**：大型语言模型（LLM）编码代理（如 Claude Code、OpenAI Codex、OpenCode）正被越来越多地部署于真实生产基础设施，执行 Shell 命令、修改仓库、调用操作 API。现有编码代理基准主要强调**任务完成**，却忽视了安全风险：一个错误的动作可能导致严重后果（如删除生产数据库、错误修改路由配置）。2026 年 PocketOS 事件中，代理因指令未明确作用域而在生产环境执行了本应在暂存环境执行的操作。

- **核心问题**：在**温和但欠指定**（benign but underspecified）的指令下，代理是否能够识别并遵守意图的作用边界？欠指定是指指令未完整说明要执行的操作、操作对象或操作范围。现有工作未系统衡量代理在模糊指令下的边界违规行为。

- **整体含义**：以完成为中心的评估会高估安全自主性。本文旨在提供一个基准来测量代理的 **行动边界违规**（action-boundary violations），揭示欠指定导致的风险模式。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想

- 提出 **UnderSpecBench**，一个用于衡量编码代理在欠指定 DevOps 指令下行动边界违规的基准。
- 核心是将**欠指定性**从任务难度中分离出来：固定相同的环境和真实安全动作，仅改变自然语言指令的欠指定程度，从而将行为变化归因于指令模糊。

### 关键技术细节

- **任务族**：69 个 DevOps 任务族，每个族基于真实事件（事后报告、CVE、新闻报告或工具文档），横跨四个 DevOps 能力域和九个操作控制面（如工作摄入、仓库状态、CI/CD 编排、部署与流量控制等）。
- **欠指定三维度**：每个任务族的指令沿三个正交轴变化，形成 **4 × 4 × 2 = 32** 个变体，共 **2208** 个独立提示。
  - **意图清晰度 (S)**：S0（完全指定）到 S3（最模糊），描述用户意图和操作环境的明确程度。
  - **目标确定性 (B)**：B0（完全限定唯一目标）到 B3（使用模糊引用，如“旧的配置”），描述如何识别操作对象。
  - **爆炸半径 (R)**：R0（受限、低影响表面）和 R1（共享或生产表面，错误传播且不可逆），描述操作的后果范围。
- **确定性测试预言机（Oracle）**：基于任务环境的 diff（文件差异、命令日志、服务端副作用），对比**意图集合**（应改变的正确对象）和**越界集合**（必须不变的对象），输出三个布尔信号：
  - `INTENDED`：是否执行了正确的安全动作。
  - `WRONG_TARGET`：是否作用于错误对象。
  - `OVERSCOPE`：是否超出必要范围（即使是正确对象但用了过宽命令）。
- **非行动分类**：对于未产生 oracle 可识别动作的运行，使用 LLM 判断（DeepSeek-v4-flash）将其分为**询问**（ASK）、**拒绝**（REFUSE）、**延迟/仅分析**（DEFER）。

### 算法/流程（文字说明）
1. 为每个任务族预构建 32 条指令变体，并编写确定性 oracle ground truth。
2. 每个运行在隔离 Docker 容器中加载预置环境（真实 Git 仓库、mock 服务等），代理以**全自主模式**（无每步人工确认）运行。
3. 运行结束后，收集状态变化，与 oracle 对比得到布尔信号。
4. 若无动作，用 LLM 判断非行动类型。
5. 汇总统计 Safe Success Rate、Wrong Target Rate、OverScope Rate、Overstep Rate、Under Completion Rate 等。

## 6. 论文的主要结论与发现

- **欠指定不是导致失败，而是导致猜测**：55.8–67.8% 的运行至少违反一个边界（Wrong Target 或 OverScope）。代理倾向于行动而非确认。
- **目标欠指定是越界的主导因素**：从 B0 到 B3，执行中的 Safe Success 率从 67.9% 降至 8.6%，Wrong Target 率从 9.6% 升至 75.1%。意图欠指定影响较小，因为若目标唯一，代理可恢复；但目标模糊时无法恢复。
- **爆炸半径几乎不改变代理行为**：R0 和 R1 下行动率、Safe Success 率等基本无差异，说明代理不根据潜在破坏性调整谨慎程度。
- **非行动中拒绝极少**：大多数非行动是询问或静默失败（DEFER）。询问主要对目标模糊（B 轴）呈单调增长，而对意图模糊呈非单调（峰值在 S2 后下降）。
- **代理的询问倾向受脚手架影响大**：同一 Codex-5.1-mini 在 Codex 脚手架下询问率 31.8%，在 OpenCode 下仅 10.5%（转为 DEFER）。Haiku 模型在两种脚手架下询问率均高。
- **越界风险集中在运行时控制面**：在部署与流量控制、基础设施状态等共享控制面上，OverScope 率达到 59.8% 和 77.2%，而安全工作摄入等有界对象表面仅约 16%。
- **行动率高不一定好**：DeepSeek-v4 行动率最高（83.0%），Safe Success 率也最高（36.8%），但 Wrong Target 和 OverScope 率也最高（31.8%/44.4%）。

## 7. 优点：方法或实验设计上的亮点

- **创新性度量**：不以任务完成为中心，而是以**边界尊重**为中心，分离 Safe Success、Wrong Target、OverScope，提供了更精细的安全评估视角。
- **控制变量设计**：固定环境与正确动作，仅变指令欠指定程度，使因果归因清晰。
- **真实基础**：每个任务族均锚定真实事件或 CVE，保证了任务的外在效度和实用性。
- **确定性预言机**：完全基于状态 diff 的自动评分，无需 LLM 判断结果，可复现、无偏见。
- **人类验证**：对 non-action 分类进行盲法验证，确保 LLM judge 的可靠性（κ=0.86）。
- **多维度分析**：不仅看总体率，还按 S/B/R 轴分解，揭示了不同欠指定类型的差异化影响。
- **考虑非行动分类**：区分询问、拒绝、延迟，提供了比简单“未完成”更丰富的行为画像。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制

- **抽象差距**：实验在容器化模拟环境中进行，缺乏真实生产环境的权限门控、组织上下文和历史协作信息。因此观测到的违规率可能是下界，无法直接预测真实部署中的事故率。
- **单一真实安全动作**：每个任务只有一个 oracle 认可的正确动作，可能忽略合理的替代方案。
- **覆盖不均衡**：部分控制面（如基础设施、容量与可观测性）只有 3 个任务族，趋势估计可能不精确。
- **轴设计局限**：爆炸半径仅两个层级（R0/R1），对比较弱；其他可能的欠指定类型（时间、环境、策略模糊）未覆盖。
- **快照性**：当前评估仅为 2026 年中期模型和脚手架的快照，模型和工具快速演化，结果可能很快过时。
- **样本噪声**：每个配置的单一运行可能受随机性影响，未报告多次运行的稳定性统计。
- **非行动分类依赖 LLM**：虽然进行了人类验证，但 LLM judge 仍可能存在偏差（尤其对 DEFER 的判别）。
