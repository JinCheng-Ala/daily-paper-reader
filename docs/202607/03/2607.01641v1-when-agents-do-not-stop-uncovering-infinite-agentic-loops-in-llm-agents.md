## 摘要
LLM智能体越来越依赖迭代执行，通过规划、工具使用、状态更新和智能体协作来完成任务。这种设计虽然实现了灵活的自动化，但也产生了一类新的失败：当反馈路径未有效约束时，智能体可能会重复执行模型调用、工具、工作流转换或智能体交接。我们称此问题为无限智能体循环（IALs）。IALs不是普通的编程循环；它们源于智能体逻辑、框架语义、运行时观察和终止机制之间的交互。此类失败可将单个请求放大为长时间运行的模型和工具执行，导致成本耗尽、模型拒绝服务、上下文增长以及重复的外部副作用。我们提出IAL-Scan，一种用于检测真实世界LLM智能体项目中IAL失败的静态分析工具。IAL-Scan将异构的智能体代码抽象为框架无关的智能体IR，构建智能体循环依赖图（ALDG）以恢复显式和框架引入的反馈路径，并检查这些路径是否能反复到达昂贵或状态增长的操作而无有效约束。我们在6,549个LLM智能体仓库上评估IAL-Scan。它报告了74个潜在发现，其中人工审核确认了47个项目中的68个IAL失败，实现了91.9%的精确率。

## Abstract
LLM agents increasingly rely on iterative execution to solve tasks through planning, tool use, state updates, and agent collaboration. While this design enables flexible automation, it also creates a new class of failures: an agent may repeatedly execute model calls, tools, workflow transitions, or agent handoffs when the feedback path is not effectively bounded. We call this problem Infinite Agentic Loops (IALs). IALs are not ordinary programming loops; they arise from the interaction between agent logic, framework semantics, runtime observations, and termination mechanisms. Such failures can amplify a single request into long running model and tool execution, causing cost exhaustion, model denial of service, context growth, and repeated external side effects. We propose IAL-Scan, a static analysis tool for detecting IAL failures in real-world LLM agent projects. IAL-Scan abstracts heterogeneous agent code into a framework independent Agent IR, builds an Agentic Loop Dependence Graph (ALDG) to recover explicit and framework induced feedback paths, and checks whether these paths can repeatedly reach costly or state growing operations without an effective bound. We evaluate IAL-Scan on 6,549 LLM agent repositories. It reports 74 potential findings, among which manual review confirms 68 IAL failures across 47 projects, achieving 91.9% precision.

## 1. 核心问题与整体含义（研究动机与背景）
- **问题定义**：LLM智能体通过迭代执行（感知-推理-行动循环）实现任务自动化，但若反馈路径未有效约束，可能陷入**无限智能体循环（Infinite Agentic Loops, IALs）**——反复触发模型调用、工具执行、工作流转换或智能体交接，导致：
  - API 成本耗尽
  - 模型拒绝服务（DoS）
  - 上下文窗口膨胀
  - 重复的外部副作用（如重复写入数据库、发送请求）
- **背景**：现有框架（LangChain、LangGraph、AutoGen、CrewAI、OpenAI Agents SDK等）虽提供`max_iterations`、`recursion_limit`等终止机制，但开发者常遗漏、误配置或将其置于反馈路径之外，使得IALs在真实部署中频发。传统静态分析工具（如CodeQL、Semgrep）无法建模智能体执行语义（如框架API边、工具调度、交接等），无法检测此类故障。
- **研究必要性**：提出首个针对IALs的专用静态分析工具IAL-Scan，填补安全与可靠性分析的空白。

## 2. 方法论：核心思想、技术细节与算法流程
### 核心思想
- 将异构的智能体代码（包括源码、框架API、配置）抽象为**框架无关的智能体中间表示（Agent IR）**，再构建**智能体循环依赖图（Agentic Loop Dependence Graph, ALDG）**，从而识别出可能无界重复的反馈路径，并检查其是否被有效边界覆盖。

### 关键技术细节
1. **Agent IR 构建**
   - 定义事实集：`ExecutionUnit`（执行作用域）、`Controller`（循环/路由/重试控制器）、`Invocation`（模型/工具/智能体调用）、`StateUpdate`（跨迭代状态更新）、`Bound`（容量/超时等边界）、`ExitRecord`（退出条件）。
   - 提取过程：解析源码AST，结合轻量级名称解析，将源事实映射为Agent IR事实。同时对框架（支持8种）进行行为建模，补充隐式调用关系（如工具调度、工作流边、智能体交接、重试路径）。

2. **ALDG 构建**
   - **节点**：从Agent IR中保留`Controller`、`Costly Invocation`、`Growing StateUpdate`、`Scope`等，共12种节点类型。
   - **边**：基于Agent IR关系推导10种边，包括控制流、调用、反馈、工作流转、工具调度、退出等。关键边类型：`LOOP_BACK`（循环回边）、`FEEDBACK`（模型/工具结果导致的反馈）、`WORKFLOW_TRANSITION`、`TOOL_DISPATCH`。
   - **循环注释**：为每个控制器节点标注是否可达昂贵操作、状态增长、退出条件来源等属性。

3. **IAL 失败检测**（算法流程）
   - **Step 1: SCC候选发现**：从ALDG中提取包含反馈边的强连通分量（SCC），过滤掉非智能体循环（如流消费者、分页、生命周期循环等）。
   - **Step 2: 控制器与边界验证**：对每个候选SCC，选择主控制器并分类（确定性/模型控制/工具控制/状态控制/混合）。检查是否存在有效边界覆盖该反馈路径（边界需作用于控制器或其所属作用域，且覆盖全部反馈路径）。有效边界包括显式的迭代上限、超时、预算、递归限制等；无效边界包括模型输出控制的退出、不覆盖所有分支的局部限制等。
   - **Step 3: 识别与LLM辅助剪枝**：对满足“存在反馈路径且无有效边界且路径含昂贵/状态增长操作”的候选报告为IAL。对模糊候选，使用LLM（GPT-5.5）进行负向过滤（仅当LLM能给出强有限边界、非智能体循环等确凿理由时移除）。LLM仅作为剪枝辅助，不参与主决策。

## 6. 主要结论与发现
- **IAL-Scan有效性**：在6,549个真实项目中成功检测出68个IAL失败，精度91.9%，覆盖全部8个框架。主要失败模式包括：重试无边界（25.0%）、工具调用迭代无边界（23.5%）、多智能体对话无边界（20.6%）、工作流循环无有效边界（13.2%）、消息重入无边界（10.3%）、运行器/委托/评估器反馈（7.4%）。
- **根本原因**：所有68个失败均因反馈路径缺少强边界，常见于工具控制的重试（41.2%）、模型控制终止（38.2%）、缺失退出条件（33.8%）、工作流循环（30.9%）、状态增长放大器（27.9%）、智能体工具重入（25.0%）。
- **影响**：95.6%的失败可能导致API成本耗尽和模型DoS；27.9%可导致上下文窗口耗尽。
- **与基线对比**：IAL-Scan真阳性覆盖（68）远高于编码助手（50）和纯LLM（23），且token消耗和时间显著更低。
- **各组件必要性**：消融实验显示，每个组件对精确率和召回率均有贡献；移除框架建模或边界覆盖导致真阳性下降；LLM剪枝大幅减少人工审查负担。

- **方法创新**：提出框架无关的Agent IR和ALDG，有效处理框架隐藏的反馈行为；结合SCC检测与边界覆盖推理，设计LLM辅助剪枝作为可选优化。

- **充分分析**：包含消融、基线对比、稳定性与敏感性实验，结果可信。
- **实际价值**：能帮助开发者在部署前发现潜在无限循环，节省成本并提升可靠性。

## 8. 不足与局限
- **静态分析过近似**：可能报告假阳性（论文中共6个），源于间接或框架依赖的边界识别困难。
- **语言与框架限制**：仅支持Python及8个流行框架，不覆盖其他语言（如JavaScript）或较少用的框架，可能导致假阴性。
- **用户定义语义支持有限**：对高度定制的调度器、外部状态驱动的停止逻辑、基于自然语言输出的语义检查无法精确建模。
- **LLM剪枝不稳定性**：端到端重复运行和模型选择均引入波动（真阳性覆盖从60到68），其可靠性依赖于模型能力；论文仅将其作为负向过滤器，但未完全消除误判风险。
- **假阴性估计不完整**：仅通过两种基线分析发现7个假阴性，可能还有其他未被覆盖的IAL模式。
- **未考虑运行时动态行为**：如动态工具注册、网络延迟、并发竞争等，这些可能触发或缓解IALs，但静态分析无法捕捉。
