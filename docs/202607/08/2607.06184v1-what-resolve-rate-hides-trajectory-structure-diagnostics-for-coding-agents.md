## 摘要
编码代理几乎完全根据解析率进行排名：其最终补丁是否通过目标测试。然而，两个代理可以通过截然不同的过程达到相同的结果，单一的通过/失败标签无法说明运行为何失败，或者成功的运行为何花费了额外的步骤、时间或令牌。这种过程证据存在于轨迹中，轨迹记录了运行的搜索、读取、编辑、工具调用、验证和回退。然而，原始轨迹是异构的且难以跨运行比较。我们提出了TraceProbe，一个轨迹诊断框架，揭示了解析率隐藏的信息。TraceProbe将每次原始运行标准化为具有确定性效果标签的九种动作类型分类，然后应用两个基于规则的模块：Insight用于识别从已建立的调试实践中改编的单轨迹反模式（例如，搜索循环、验证跳过），而Converge则对齐成对运行并分类它们在受控参考下的行为分歧点。将TraceProbe应用于SWE-Bench Verified上五个生产环境的2500条轨迹，我们发现：(i) 文件选择过于粗糙，无法区分成功与失败，而函数选择和完成行为则能定位问题；(ii) Insight反模式主要作为语料库级别的难度线索，其中搜索循环最为稳定；(iii) 即使是已解决的运行，在达到相关代码的速度和产生的失败工作量上也存在差异。因此，轨迹结构通过定位检查目标、提出失败假设和优先安排运行审查，为结果增加了可审计的诊断上下文。

## Abstract
Coding agents are ranked almost entirely by resolve rate: whether their final patch passes the target tests. Yet two agents can reach the same outcome through very different processes, and a single pass/fail label says nothing about why a run failed or why an accepted run spent extra steps, time, or tokens. This process evidence lives in the trajectory, which records a run's searches, reads, edits, tool calls, validation, and reversions. However, raw traces are heterogeneous and hard to compare across runs. We present TraceProbe, a trajectory-diagnostic framework that recovers what resolve rate hides. TraceProbe normalizes each raw run into a canonical nine-type action taxonomy with deterministic effect labels, then applies two rule-based modules: Insight names single-trajectory anti-patterns adapted from established debugging practice (e.g., search loops, verification skips), while Converge aligns pairs of runs and classifies where their behavior diverges under controlled references. Applying TraceProbe to 2,500 trajectories from five production settings on SWE-Bench Verified, we find that (i) file choice is too coarse to separate success from failure, whereas function selection and completion behavior localize it; (ii) Insight anti-patterns act mainly as corpus-level difficulty clues, with search loops the most stable; and (iii) even resolved runs differ in how quickly they reach relevant code and how much failed work they incur. Trajectory structure thus adds auditable diagnostic context to outcomes by localizing inspection targets, suggesting failure hypotheses, and prioritizing runs for review.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：当前编码代理（Coding Agent）的性能评估几乎完全依赖**解析率（Resolve Rate）**——即最终补丁是否通过目标测试。然而，这一单一指标隐藏了大量**过程信息**：两个代理可能通过截然不同的路径达到相同结果，而一个失败运行的具体原因、成功运行中多余步骤或令牌消耗等细节均被掩盖。
- **研究动机**：原始轨迹记录（包括搜索、读取、编辑、工具调用、验证和回退）包含丰富的诊断证据，但不同代理的轨迹异构性强、难以直接比较。现有工作或只关注最终结果（如SWE-Bench），或提供聚合统计/分数，缺乏**确定性、可审计、命名模式**的过程诊断工具。
- **整体含义**：提出**TraceProbe**框架，通过将原始轨迹标准化为规范动作序列，并应用基于规则的检测和对齐模块，**恢复被解析率隐藏的过程差异**，为开发者提供可指导检查、失败假设和审查优先级的具体证据。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将原始异构轨迹转换为**共享规范表示**，再通过确定性子模块进行**单轨迹反模式检测**和**跨轨迹行为对齐与分类**，从而实现细粒度、可复现的过程诊断。
- **关键技术细节**：
  1. **共享表示构建**：
     - 通过**支架适配器（Scaffold Adapters）**将不同代理（Claude Code、Codex、OpenCode）的原始JSON轨迹标准化为统一步骤。
     - 每个步骤映射为**九类规范动作**：FILE READ, FILE WRITE, SEARCH, COMMAND, PLAN, NAVIGATE, FETCH, AGENT SPAWN, REASON。
     - 为每个动作赋予**确定性效果标签**（SURVIVED, FAILED, REVERTED, JUSTIFIED, RECORDED, OFF-ANCHOR, REASONING），基于可观察状态转换，无需LLM判断。
     - 将动作归类为**工作流阶段**（理解、计划、实现、调试、验证、报告）。
     - 对于基准任务，利用**锚定集（Gold-patch changed files）**标记任务相关文件/函数，支持离锚（off-anchor）标签和里程碑提取。
  2. **Insight模块**：单轨迹分析
     - 定义一组**确定性反模式检测器**，每个检测器有精确谓词和固定阈值（如搜索循环、重读搅动、工具振荡、无正式尾验证、不支持完成声称等）。
     - 检测器基于规范动作序列和效果标签，无需外部或acles。
     - 另有**语义检测器**（使用LLM辅助标签）作为探索性分析，不用于主要结论。
  3. **Converge模块**：跨轨迹分析
     - 采用**最长公共子序列（LCS）**算法对齐参考轨迹与比较轨迹，动作匹配基于类型、目标和效果标签兼容性。
     - 将未匹配或重排的动作分组为**分歧跨度（divergence spans）**，并通过固定阈值评估三个诊断维度：**文件选择**、**编辑稳定性**、**完成行为**。
     - 每个维度输出强/中/弱一致等级，表示与参考轨迹的偏离程度。
  4. **里程碑提取**：记录轨迹中最早达到锚定文件读取/写入/全部写入/首次通过验证/首次合理动作的步数，用于衡量任务相关进展时机。

## 6. 主要结论与发现

1. **文件选择过于粗糙**：成功运行和失败运行可能接触相同文件，而**函数选择**和**完成行为**能更有效地定位失败点。跨度分析显示，失败运行通常在文件范围上偏离参考，中位函数级别分歧出现在失败前约20步，足以指导检查。
2. **反模式主要作为语料库级难度线索**：
   - **搜索循环（Search Loop）**是最稳定的失败关联模式，在两个基准上都保持低于1的胜负比（更多出现在失败运行中）。
   - **重读搅动（Re-read Churn）**关联较弱且对数据集敏感；其他模式如工具振荡、无正式尾验证等要么弱要么向解析运行偏移。
   - 大部分反模式随任务难度增加而增加，因此简单聚合可能混淆任务难度效应与特定过程信号。
3. **成功运行仍存在过程差异**：
   - 使用**效果标签**（存活、失败、回退、合理）可以揭示不同设置的过程特征。例如Codex具有最高的存活动作比例；OpenCode/GPT-5.4具有最高的失败动作比例；OpenCode/Opus具有最低的离锚比例和最高的合理动作比例。
   - **里程碑**（首次锚写入时机）显示，即使同为已解决运行，不同设置的进展时机差异显著（如OpenCode/Opus最早达到首次锚写入，而Claude Code和Codex更晚）。归一化后，OpenCode/GLM虽总步数较长但早期即触及相关代码。
4. **参考范围的比较稳健且受模型影响**：
   - 在相同模型对比中，GPT-5.4下两种支架（Codex vs OpenCode）在文件选择上高度分歧（93%弱一致），而Opus 4.6下相同支架对比的分歧较小。
   - 参考依赖性检验：文件选择几乎不受参考选择影响（仅0.69%变化），编辑稳定性对参考较敏感（22.1%变化），完成行为居中（8.7%）。
5. **跨基准迁移有限定**：搜索循环的胜负比在两个基准上均稳定低于1，但具体发生率差异大；其他检测器的方向或关联强度在Pro上发生变化，说明阈值在不同基准上需重新审计。

## 7. 优点

- **确定性、可审计性**：所有主要检测器和效果标签基于规则，无LLM判断，结果可完全复现；阈值固定，避免主观偏差。
- **共享规范化表示**：使不同支架、模型的轨迹可统一比较，解决了异构性问题。
- **命名模式体系**：将软件工程中已知的反模式（如搜索循环、工具振荡）转化为精确的轨迹谓词，提供直观诊断语言。
- **多层级分析**：覆盖单轨迹（反模式）、跨轨迹（分歧标准）、进展时机（里程碑）和过程成本（效果标签比例），视角全面。
- **控制严谨**：通过相同任务、相同结果、固定参考、重复运行等设计隔离混杂因素，实验可靠性高。
- **可迁移性验证**：在两个独立基准（Verified和Pro）上验证检测器行为，明确说明哪些信号稳定、哪些需要重新审计。

## 8. 不足与局限

1. **基准特异性**：主要实验在SWE-Bench Verified上校准阈值，部分检测器（如重读搅动）在Pro上关联减弱，需在每个目标基准上重新审计阈值。
2. **任务类型限制**：仅覆盖Python问题到PR修复任务（issue→patch），结果可能不泛化到其他语言（如Java、C++）或任务类型（如代码生成、漏洞修复）。
3. **单次运行限制**：除稳定性子样本外，主实验每个任务/设置只运行一次，未控制运行间噪声；虽有子样本检验，但未覆盖全部500任务。
4. **缺乏因果解释**：TraceProbe提供关联证据和定位线索，但不能证明某个反模式是失败原因；作者明确声明不是每运行根因解释器。
5. **未评估实践影响**：未测量诊断信号能否实际改善开发者决策效率或代理调试效果。
6. **语义检测器局限**：虽然主要结果基于确定性检测器，但语义检测器（如阶段振荡、语义无用探索）依赖LLM标签，可能受标签模型变化影响，仅作探索使用。
7. **执行测量依赖环境**：持续时间、令牌数、成本等受CLI版本、提供商批处理、速率限制等影响，不能视为模型固有效率。
