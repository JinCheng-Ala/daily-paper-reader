## 摘要
我们提出了AgentLens，一个面向交互式代码智能体的生产评估基准。大多数代码智能体基准将一次运行简化为单个比特——任务是否通过？但实际使用这些智能体的人员体验的是整个轨迹：智能体如何遵循指令、使用工具、验证自身工作、从错误中恢复，以及在此过程中与他们交流。AgentLens评估整个轨迹。它将存在客观检查的形式化验证与LLM编写的轨迹审查及并排比较相结合，从而每次运行都能得到一个可读的解释，说明得分为何如此。这使得AgentLens不仅可用于模型排名：我们用它来诊断模型行为，比较我们自身智能体的连续版本，并在夜间评估流水线中捕捉产品回归问题。我们以开源形式发布该基准，地址为https://github.com/agent-lens/agent-lens-bench。

## Abstract
We present AgentLens, a production-assessed benchmark for interactive code agents. Most code-agent benchmarks reduce a run to a single bit -- did the task pass? -- but the people who actually use these agents experience the entire trajectory: how the agent follows instructions, uses its tools, verifies its own work, recovers from mistakes, and talks to them along the way. AgentLens evaluates that whole trajectory. It pairs formal verification, where an objective check exists, with LLM-written trajectory reviews and side-by-side comparisons, so that each run yields a readable explanation of why the score is what it is. This makes AgentLens useful for more than ranking models: we use it to diagnose model behavior, compare successive versions of our own agent, and catch product regressions in a nightly evaluation pipeline. We release the benchmark as open source at https://github.com/agent-lens/agent-lens-bench.

## 1. 核心问题与整体含义（研究动机和背景）

*   **核心问题**：现有的代码智能体（coding agent）评估基准（如SWE-bench）通常只用一个二进制位（pass/fail）来衡量一次运行是否成功，完全忽略了用户在实际使用中体验到的**完整交互过程**。
*   **背景与动机**：开发者在使用编码助手时，不仅关心最终结果是否正确，更关心智能体如何遵循指令、使用工具、验证自身工作、从错误中恢复以及与用户的沟通质量。一个在狭窄检查中“通过”的运行可能仍然不可靠、过度自信、交互体验差或与用户实际意图有细微偏差；而一个“失败”的运行可能在此过程中做了很多有用的事。例如，编写项目文档的任务根本无法用二进制结果衡量。AgentLens 旨在填补这一空白，通过评估**完整轨迹**来提供更贴近用户实际感知的度量。

## 2. 方法论

*   **核心思想**：将评估对象从**最终状态**转向**完整轨迹**。轨迹包括用户消息、智能体回复、工具调用、文件编辑、命令执行、验证尝试、最终回复以及仓库最终状态。采用“形式化验证 + LLM 评委审查”的混合协议，使得每项评分都附带可读的证据解释。

    *   **任务集与运行**：初始发布包含 16 个编码智能体场景（workflows），每个场景使用默认和有毒（toxic）两种用户角色（由LLM模拟），共 32 条轨迹。任务源自开发者访谈和匿名的产品使用记录。
    *   **轨迹审查协议**：
        *   **形式化验证**：提供客观检查，包括仓库状态检查、正则表达式检查、测试执行、命令输出检查、静态分析等（共 10 种验证器，见表1）。

## 6. 主要结论与发现

*   AgentLens 能够**捕获二进制指标遗漏的用户可见质量维度**，其质量指数（QI）与公共基准中的智能体类基准（如APEX-Agents）相关性中等（ρ=0.82），但与推理/指令遵循类基准几乎无关甚至负相关（如IFBench ρ=-0.41，τ2-Telecom ρ=-0.25），表明评估维度独特。
*   **模型行为差异显著**：
    *   Gemini 3.1 Pro Preview 的最终排名靠前但仍明显落后于前沿模型，原因是**代理循环可靠性不足**（工作流控制、验证失败、编辑循环），而非代码生成能力弱。
    *   DeepSeek V4 Pro 与 Flash 的 QI 接近，但呈现**清晰的质量权衡**：Pro 在指令遵循和安全性上更好，Flash 因更高吞吐量在完成度上更优。
    *   **Opus vs GPT-5.5**：GPT-5.5 更擅长遵循任务形式（步骤、消息格式），Opus 更擅长保留任务意图和语义可靠性。
*   **定性审查比单纯分数更具诊断价值**：通过具体案例展示了如何识别提供商 bug（Kimi 的 JSON 参数解析错误）和回归（并行工具调用中的并发修改异常）。
*   **基准适用于产品级回归检测**：增强了开发流程中捕捉回归的能力。

## 7. 优点

*   **轨迹级评估**：关注完整交互过程而非单一结果，更贴近实际使用体验。
*   **可解释性**：每个得分都附带基于证据的书面审查，便于诊断具体失败模式。
*   **多维度覆盖**：同时评估结果、过程、交互质量、工具使用等，分离表现因素。
*   **形式化验证与主观判断的结合**：利用客观检查抵消LLM评委的偏差，同时弥补狭窄检查可能遗漏的奖励黑客行为。
*   **并排比较**：支持细粒度模型对比、特征评估和回归检测。
*   **生产导向**：直接集成到 CI/CD 流水线中进行夜间回归检测，具有实际工程价值。
*   **开源性**：代码和基准集开源，便于社区扩展和验证。

## 8. 不足与局限

*   **任务集规模小且仅限 Java**：当前仅包含 16 个场景（32条轨迹），语言限制影响结论的通用性。
*   **LLM 评委的可靠性证据不足**：仅凭内部小规模非正式实验声称人机一致性尚可，缺乏正式的人工一致性研究和跨模型法官的鲁棒性测试（法官自我偏好偏差约4%）。
*   **第三方提供商不可控**：部分模型通过 OpenRouter 等 API 访问，延迟、路由、版本和故障（如 Kimi 案例）不属于论文控制范围。
*   **评估成本较高**：单次完整运行可能超过 100 美元（使用顶级模型），限制了大规模重复实验。
*   **系统偏见风险**：基准最初围绕自研的 Explyt AI Agent 构建，某些任务或集成假设可能对其更有利（虽然通过开源部分缓解）。
