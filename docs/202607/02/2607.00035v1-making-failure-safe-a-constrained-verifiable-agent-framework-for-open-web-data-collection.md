## 摘要
大语言模型和智能体可以从自然语言需求生成网络爬虫，但直接生成仍然不可靠，因为存在依赖错误、选择器损坏、模式不匹配和页面结构异质等问题。我们提出一个受限、可验证的智能体框架，将LLM输出从自由形式代码转变为类型化的JSON收集器配置，结合了六种类型的收集器分类法、模板和实用函数约束、静态Airflow DAG执行、基于规则的质量检查以及结构化反馈修正。在138个任务上的实验表明，该分类法支持基于描述的需求类型化，同时确认稳定实例化需要完成超出初始描述的源、字段和执行约束。在80个独立源验证的任务上，该框架以零执行阶段LLM令牌和最低平均挂钟时间运行，以适中的一次性质量换取适合重复计划收集的可复用、确定性和可验证的执行路径。这些结果将该框架定位为用于重复开放网络数据收集的可复用、低成本且可验证的执行路径。

## Abstract
LLMs and agents can generate web scrapers from natural-language requirements, but direct generation remains unreliable because of dependency errors, broken selectors, schema mismatches, and heterogeneous page structures. We propose a constrained, verifiable agent framework that shifts LLM output from free-form code to typed JSON collector configurations, combining a six-type collector taxonomy, template and utility-function constraints, static Airflow DAG execution, rule-based quality checking, and structured feedback correction. Experiments on 138 tasks show that the taxonomy supports description-based requirement typing, while confirming that stable instantiation requires completing source, field, and execution constraints beyond the initial description. On 80 independently source-verified tasks, the framework runs with zero execution-stage LLM tokens and the lowest average wall-clock time, trading moderate one-shot quality for a reusable, deterministic, and verifiable execution path suited to repeated scheduled collection. These results position the framework as a reusable, low-cost, and verifiable execution path for repeated open-web data collection.

## 2. 论文提出的方法论

### 核心思想
通过**四层嵌套约束**将LLM输出从自由形式代码转变为类型化JSON配置，形成“生成–执行–检查–修复”闭环。这四层约束是：
- 任务类型约束（确定收集器类型）
- 配置结构约束（模板槽位 + JSON Schema）
- 工具复用约束（预建实用函数，不生成请求/解析/清洗代码）
- 执行反馈约束（小样本验证 + 质量评估 + 结构化反馈修正）

### 关键技术细节
1. **六类收集器分类法（Six-Type Collector Taxonomy）**：
    - Search（关键词搜索发现URL）、List（分页列表遍历）、Detail（结构化字段提取）、API（调用公共API）、Interactive（动态页面交互）、File（下载解析PDF/Excel）。遵循单职责、明确边界、调度器可组合等原则。
2. **需求理解智能体（Requirement Understanding Agent）**：
    - 接受自然语言描述 `d`，通过主动探测目标站点（主页、导航、robots/sitemap、API文档、样本页）获取源上下文 `x`，生成完整任务表示 `R = (d, s, x, f, c, o)`。输出被约束到固定Schema，包含类型槽、置信度等。
3. **收集器实例化智能体（Collector Instantiation Agent）**：
    - 采用“参考编码 + 模板槽填充”：模板硬编码Airflow结构、实用函数骨架、错误处理、持久化；智能体填入URL、选择器、API参数、字段映射、清洗规则。输出JSON配置 `M`（非可执行代码），由通用收集器操作符解释执行。
    - 提示上下文包括：系统提示（类型规范、Schema约束）、少样本示例、完整任务表示 `R`、槽填充指令。联合约束确保 `θ* ∈ Θk` 且 `Check(M) = True`。
4. **Airflow验证与规模执行**：
    - 使用静态执行DAG，通用收集器操作符读取JSON配置并调用预建实用函数。验证阶段使用小样本参数（限制页面、输出行数、强制原始页面存档），规模阶段使用生产参数。LLM生成的代码不直接进入DAG，避免破坏Airflow调度器解析。
5. **质量检查模块（Quality Check）**：
    - 基于规则的确定性检查（无LLM），评估五个维度并计算复合分数 `S = ws·Ss + wc·Sc + wr·Sr + wa·Sa + we·Se`，默认权重 (0.35,0.25,0.20,0.10,0.10)。维度包括：结构（有效字段比例）、内容（非空值比例）、相关性（关键词重叠相似度）、附件（有效附件比例）、执行（错误率）。质量报告包含各维度分数、通过/失败判定、错误标签、原始页面对比（区分源缺陷与解析错误）、执行元数据、修正提示。
6. **反馈修正模块（Feedback Correction）**：
    - 将质量报告 `Q` 转换为反馈约束 `Γt`（需修复的槽、禁止的假设、候选修正方向）。维护失败黑名单 `Bt`（已失败的选器、字段路径、假设）。修正优先级顺序：解析错误 < Schema错误 < 运行时错误 < 内容噪声 < 低相关性。算法1是状态机核心逻辑，当无可用候选时触发人工审核。

## 6. 论文的主要结论与发现

1. **需求类型分类**：在仅自然语言描述下，Ours的Macro-F1达到0.7793，高于关键词基线0.7562，但精确匹配均为0.7101，说明描述不足的分类有歧义（尤其Detail和List任务）。
2. **收集器实例化**：Ours达到最高平均质量0.5369、通过率0.5507、类型匹配率0.8551。无约束LLM生成虽然通过率0.5217，但类型匹配率仅0.1884（自由生成保留了字段但不维护类型语义）。规则模板基线仅0.3184质量。
3. **质量-成本权衡**（80源验证任务）：
    - 运行时LLM基线（Schema-constrained LLM、Crawl4AI+LLM）取得更高一次性质量（0.7051和0.6996）和通过率（>0.80）。
    - Ours质量较低（0.5631，通过率0.5000），但**零执行阶段LLM Tokens**，平均墙钟时间最低（5.052秒，对比基线14.257–20.194秒）。
    - 结论：Ours适合**重复调度采集**，将LLM成本摊薄到一次性配置生成；运行时LLM更适合单次高质量提取。
4. **反馈修正**：从0.5087质量/0.0072通过率提升至0.9007质量/1.000通过率，平均修正轮次1.99，证明对已知缺陷修复有效。
5. **工程验证**：全部138个Ours配置都复用共享模板和工具（复用率1.0000 vs. 无约束生成0）；80任务运行成功率80.0%；静态DAG通过Airflow 2.8.0完整测试。
6. **研究问题回答**：
    - RQ1：类型化建模有效（Macro-F1 0.7793），但需要源探测和字段确认。
    - RQ2：约束实例化提高类型匹配率和质量，但一次性质量低于运行时LLM。
    - RQ3：验证+反馈修正能力被证实（质量提升显著）。

## 7. 优点

- **创新约束范式**：将LLM从自由代码生成转变为受类型-模板-工具-反馈四层约束的JSON配置生成，从源头减少不可靠性。
- **可验证性**：每个阶段可审计，质量检查基于规则（无LLM主观判断），失败成为可追踪信号。
- **成本效率**：执行阶段零LLM Tokens，适合重复采集；配置生成的一次性成本可摊薄。
- **工程实用**：直接兼容Airflow调度，静态DAG避免LLM代码破坏管道；模板和工具复用率高。
- **全面实验**：覆盖分类、实例化、质量-成本、反馈修正、工程验证，使用统一LLM后端，对比方法多样。
- **反馈修正机制设计成结构化约束**（而非自然语言反思），避免LLM自检失败问题，且包含黑名单单调剪枝。
