## 摘要
全面测试对于自动驾驶系统（ADS）的安全性和可靠性至关重要。现有技术能够检测系统级故障或将故障归因于粗粒度模块，但通常无法在源代码中定位根本原因。因此，调试仍然需要大量人工，要求开发者将行为违规与复杂的实现逻辑联系起来。为弥补这一不足，我们提出了HINT，一个基于假设验证和意图分析的两阶段层次化ADS故障定位框架。在第一阶段，HINT将触发故障的执行记录转换为多模态抽象，并使用因果推理来识别责任模块。在第二阶段，它重构设计侧意图和实现侧行为，然后通过可靠性感知一致性检查定位可疑代码，无需昂贵的重新模拟。我们在Apollo上跨多种故障模式和模块评估了HINT。结果表明，HINT在模块级诊断和代码级定位指标上实现了最强的整体性能，在真实世界漏洞上的端到端Class@5准确率达到77.8%。

## Abstract
Comprehensive testing is essential for the safety and reliability of Autonomous Driving Systems (ADS). Existing techniques can detect system-level failures or attribute them to coarse-grained modules, but they often fall short of localizing the root cause in source code. As a result, debugging remains labor-intensive, requiring developers to connect behavioral violations with complex implementation logic. To address this gap, we present HINT, a two-phase framework for hierarchical ADS fault localization based on hypothesis validation and intent analysis. In Phase I, HINT transforms failure-triggering execution recordings into multi-modal abstractions and uses causal reasoning to identify the responsible module. In Phase II, it reconstructs design-side intent and implementation-side behavior, then localizes suspicious code through reliability-aware consistency checking, without costly re-simulation. We evaluate HINT on Apollo across diverse failure modes and modules. The results show that HINT achieves the strongest overall performance across module-level diagnosis and code-level localization metrics, with 77.8% end-to-end Class@5 accuracy on real-world bugs.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：自动驾驶系统（ADS）是高安全关键系统，需要通过大量仿真测试确保可靠性。现有测试技术能够检测系统级故障（如碰撞、交通规则违反、任务未完成），或将故障归因到粗粒度模块（如规划、感知等），但无法定位到源代码级别的根本原因。这导致调试过程仍高度依赖人工，开发者需要将行为违规与复杂实现逻辑联系起来，效率低下。
- **背景挑战**：
  - ADS故障常表现为行为偏差而非代码崩溃或异常消息，难以连接观察到的故障与责任实现逻辑。
  - 模块级故障报告不足以定位，因为执行路径组合爆炸。
  - 语义错误（语法正确的代码逻辑有误）会系统性地误导自动调试和人工分析。
- **研究目标**：提出HINT，一个两阶段层次化框架，自动从运行时记录、ADS文档和源代码中定位代码级故障根因。

## 2. 方法论

### 核心思想
HINT采用两阶段流水线：第一阶段（Phase I）将故障触发记录转化为多模态抽象，通过因果推理识别出责任模块；第二阶段（Phase II）重构设计侧意图和实现侧行为，通过可靠性感知的一致性检查定位可疑代码段，无需昂贵的重新模拟。框架依赖ADS特定证据（场景测试结果、BEV重建、STL故障规格），但可迁移到其他模块化系统。

### 关键技术细节

#### Phase I: 模块错误诊断
- **步骤1：场景抽象**：将原始运行时记录解析为结构化的时间对齐执行轨迹；使用STL监视器精确定位违反区间（violation interval Δv）；通过BEV重建叠加地图元素与轨迹；使用多模态大语言模型（MLLM）综合BEV图像和信号数据生成情境描述。
- **步骤2：假设生成与验证**（算法1）：
  - 从观察到的症状和违反区间初始化假设集合H。
  - 迭代选择最可能的假设h，映射到合适的验证技能（Skill），技能通过查询轨迹和场景产生有依据的观察。
  - 验证功能是结构化的、依赖性和语义感知的检查，判断候选模块行为是否与运行时证据一致（可解释、与下游行为一致、在语义上合理）。
  - 基于验证结果进行因果责任分析：判断不一致是源故障、传播效应、补偿响应还是偶发异常。若证据矛盾则修剪假设；否则更新局部故障元组Elocalized = ⟨m, b, c⟩（怀疑模块、违反行为/故障类型、支持证据）。

#### Phase II: 代码级故障定位
- **双流意图不一致分析**：
  - 首先构建动态执行骨架（DES）：一种场景特定的有序任务序列（任务如Apollo中的decider、optimizer、predictor等），通过识别入口锚点、静态依赖和框架执行规范，结合场景证据进行运行时感知裁剪，得到任务列表和代码搜索边界。
  - **目标意图提取（设计侧）**：由Architect代理执行，使用多阶段RAG从设计文档中检索相关证据，必要时进行自纠正/推理补充隐含约束，最终生成意图四元组I_target = ⟨Goal, Guard, Safety, Assumption⟩（结构化的自然语言）。
  - **实现意图提取（实现侧）**：由Diagnoser代理执行，通过构建语义树（由入口点开始，选择性展开行为相关的函数调用），进行逻辑抽象和收敛判断，直至实现意图I_impl不再受未解析依赖影响。
  - **不一致分析**：对每个任务进行字段级比较（Goal, Guard, Safety, Assumption），计算不一致强度V(t) = max_d (λ_d * δ_d(t))，其中λ_d为字段权重（Safety最高）。可靠性因子R_g(t)（设计证据质量）和R_c(t)（代码提取完整性）用于校准，最终怀疑分数S(t) = p0 + (V(t) - p0) * R_g(t) * R_c(t)，其中p0为均匀基线。高于阈值τ的任务保留，分数传播到函数级别。

## 3. 实验设计

### 数据集/场景
- **构建了两个互补数据集**：
  - Dinject（注入故障）：63个语义故障（Control 20, Planning 30, Prediction 28），包括反转守卫条件、缺失安全检查、错误阈值等。
  - Dreal（真实故障）：9个从Apollo公开commits和pull requests复现的历史bug（Control 1, Planning 5, Prediction 3）。
  - 所有用例涵盖三类STL oracle：碰撞、交通规则违反、任务未完成。故障触发到故障呈现的延迟不超过5秒。
- **基准系统**：Apollo 9.0 + Carla 0.9.14，Ubuntu 22.04。

### 对比方法
- Phase I（模块级诊断）：
  - ACAV：从事故记录推断因果事件。
  - Oracle-based：基于MoDitector思想，当模块输出违反特定oracle时标记。
- Phase II（代码级定位）：
  - LocAgent：基于图的仓库导航。
  - AutoFL：用LLM探索函数和调用关系。
  - Claude Code：代表性仓库级代码代理。
  - 注：Phase II对比时均提供相同的模块上下文（真实故障模块标注），以隔离Phase II效果。
- 端到端评估：
  - Claude Code：直接接收故障记录/STL报告、设计文档和仓库。
  - Claude Code + Skills：额外提供HINT的诊断技能脚本。
  - 均不使用HINT的选模块、DES或意图分析输出。

### 评估指标
- Phase I：模块级准确率。
- Phase II与端到端：Top-k准确率（Acc@k），包括Class@1/3/5和Function@3/5/10。对于函数级，使用最大得分（若函数出现在多个任务中）。

## 6. 主要结论与发现

- **模块级诊断**：HINT在Dinject和Dreal上准确率分别为88.9%和100%（平均），远超ACAV和Oracle-based基线（例如基线在碰撞类型上最好约74%）。混淆矩阵显示对角线主导，错误主要发生在功能相邻模块之间。
- **代码级定位**：HINT在函数级和真实漏洞上表现最佳。在Dinject上Class@5达74.6%，Func@10达90.5%；在Dreal上Class@5达88.9%，Func@10达88.9%。Claude Code在Class@1上略高（57.2% vs 46.0%），但HINT在大Top-k和函数粒度上更可靠。
- **消融研究**：去除场景抽象导致Phase I准确率从88.9%降至33.3%，步骤数增至15+，超时率50%。去除DES使Phase II Class@1降至0%。去除不一致分析也使Class@1降至0%，Func@3仅11.1%。证实三个组件均不可或缺。
- **端到端性能**：HINT在Dinject上Class@5为66.7%，Func@10为81.0%；在Dreal上Class@5为77.8%，Func@10为77.8%。优于Claude Code和Claude Code+Skills。Claude Code+Skills在Dinject上提升明显，但在Dreal上无提升，说明脚本不足以独立工作。
- **鲁棒性**：在不同LLM backbone上，Class@5稳定在66.7%-77.8%，Func@5在44.4%-66.7%之间。Sonnet 4.5最佳。结构化的两阶段流水线是增益的主要原因，而非特定模型。

## 7. 优点

- **首次自动化代码级故障定位**：针对ADS，从运行时记录、文档和代码直接定位到函数级根因，无需重新模拟，实用性高。
- **层次化设计合理**：先模块级诊断缩小范围，再代码级精细定位，有效缓解搜索空间爆炸。
- **创新性结合因果推理和意图不一致分析**：Phase I使用假设验证和因果责任归属区分源故障与传播效应；Phase II通过设计意图与实现意图的结构化比较（四元组）定位语义错误，并引入可靠性校准。
- **强实验评估**：构建了多故障类型的基准（包括真实bug），对比当前最强基线（ACAV、LocAgent、AutoFL、Claude Code等），进行消融和鲁棒性分析，结果全面且可复现（提供了复制包）。
- **模块化设计与可迁移性**：框架逻辑与ADS特定证据解耦，可推广到其他模块化系统（如Autoware），只要具备运行时证据、文档和代码。

## 8. 不足与局限

- **实验覆盖**：
  - 真实漏洞数据集（Dreal）仅9个用例，规模较小，可能影响统计显著性和泛化性。作者指出这反映了公开可复现的Apollo安全/性能bug的自然稀缺性。
  - 仅覆盖了Control、Planning、Prediction模块，未涉及Perception模块（需要可复现的感知故障和可靠标注）。针对端到端神经驾驶系统等不同架构不适用。
  - 仅处理故障触发到呈现延迟≤5秒的case，更长延迟的故障未验证。
- **内外部效度威胁**：
  - 依赖LLM推理质量：虽然通过结构化提示和约束工作流缓解，但仍存在模型偏差。实验表明结构比模型更重要，但未测试更弱的模型（如小参数开源模型）。
  - 假设每个故障只有一个主导故障模块，可能简化了多模块级联故障或共享因果链上的相邻函数的问题。
  - HINT依赖设计文档的完整性和源代码的可导航性；若文档不完整或代码组织不同（如Autoware），迁移可能需要调整适配器。
  - Phase II使用固定的统一意图模式（Goal, Guard, Safety, Assumption），可能不完全适用于所有任务或设计文档不足的情况，届时需求助于推断，增加不确定性。
- **评估约束**：
  - 代码级定位采用函数级而非语句级，因为ADS逻辑缺陷常跨多个谓词、分支和辅助调用。但函数级粒度可能不够细，不利于直接修复。
  - 与现有基线对比时，隔离Phase II的方法获得了真实的模块标注，这并非完整流水线情形；但端到端实验弥补了这一点。
- **资源与可重复性**：LLM调用成本未报告，实际部署时可能较高（多次调用Sonnet/Haiku）。复制包虽提供，但依赖闭源LLM（Claude, GPT）可能限制完全复现。
