## 摘要
大多数使用工具的LLM代理的评估都假设所有工具正常工作。当工具超时、返回过时值或在部署过程中其描述被篡改时，开发人员需要一种可控的方式复现故障、测试修复并在部署前确认修复有效。我们提出AgentCheck，一个开源Web工作台，它将MCP服务器转变为干预界面。AgentCheck使代理在其真实工具上运行并记录每个工具响应，然后通过故障（12种类型）注入器扰动响应后重新运行代理。匹配的工具调用从缓存中重放，而在代理产生分歧后，后续工具调用则实时进行。这形成了一个复现-干预-确认循环：开发人员切换缓解措施，针对相同的故障重新运行，并观察故障是否消失。评分分为两部分：确定性通过/失败规则，以及用于解释性标签的LLM评判器，该评判器已针对人类注释进行了验证。在五个代理中，最佳代理通过了105/120个场景，最弱的只通过了77个。失败通常是无声的、自信地使用错误的工具输出而非崩溃。在最弱的代理上，重试缓解措施将超时错误故障的成功率从低至30%提高到100%，而陈旧数据故障无论采取何种缓解措施，成功率仍保持在每10次中有3-4次。AgentCheck使这些故障模式在部署前可复现、可比较和可验证。

## Abstract
Tool-using LLM agents are mostly evaluated assuming all tools work. When a tool times out, returns a week-stale value, or has its description poisoned in deployment, the developer needs a controlled way to reproduce the failure, test a fix, and confirm the fix worked before deployment. We present AgentCheck, an open-source web workbench that turns an MCP server into an intervention surface. AgentCheck runs an agent against its real tools and records every tool response, then re-runs the agent with the response perturbed by a fault (12 types) injector. Matching tool calls are replayed from cache, and later tool calls go live after the agent diverges. This yields a reproduce-intervene-confirm loop: the developer toggles a mitigation, re-runs against the identical fault, and sees if the failure goes away. Scoring has two parts: deterministic pass/fail rules, plus an LLM judge for interpretive labels, validated against human annotations. Across five agents, the best passes 105/120 scenarios and the weakest only 77. The failures are usually silent, confident use of incorrect tool outputs rather than crashes. On the weakest agent, a retry mitigation raises success on timeout error faults from as few as 30% of cases to 100%, whereas stale-data faults remain near 3-4 of 10 regardless of the mitigation. AgentCheck makes these failure modes reproducible, comparable, and verifiable before deployment.

## 一、核心问题与整体含义（研究动机和背景）

- **问题**：现有工具使用型 LLM 代理（Agent）的评估大多假设所有工具正常工作，但在真实部署中工具可能发生超时、返回过时数据、描述被投毒等故障。开发人员缺乏一种可控方式来复现故障、测试修复并进行部署前验证。
- **目标**：提出 AgentCheck——一个开源 Web 工作台，将 MCP 服务器变为干预界面，实现“复现-干预-确认”循环，使故障可复现、可比较、可验证。
- **整体含义**：弥补了任务成功率基准（如 GAIA、SWE-bench）只测理想条件、与真实部署脱节的空白；也弥补了诊断工具（如 LangSmith、Langfuse）只能观测而不能主动注入故障的不足。

## 二、方法论：核心思想、关键技术细节

### 核心思想
- 利用 MCP 服务器作为干预面，在真实工具响应上注入 12 种故障，通过缓存重放控制变量，将 clean run 与 faulted run 进行直接比较。
- 形成“复现 → 干预 → 确认”闭环：开发者对同一故障应用缓解措施，再运行一次，验证故障是否消除。

### 关键技术细节
1. **受控比较（Algorithm 1）**
   - **Clean run**：记录 agent 在真实工具上的每一次工具调用及其响应，缓存至 K。
   - **Faulted run**：使用 `REPLAY` 函数重放缓存响应，但将指定位置 (p, g) 的响应替换为故障注入后的响应；匹配的调用从缓存重放，后续调用（agent 行为分歧后）实时进行。
   - **Mitigated run**：对 agent 应用缓解包装器（如重试、模式检查等），再次用相同故障重放。
   - 输出：首次分歧点（DIVERGE）、失败的主要检查（F）、以及 `fix_confirmed`（所有失败检查在缓解后是否全部通过）。

2. **故障注入引擎**：12 种故障类型，分为三类（附录 A）：
   - **A 类 - 工具执行**：超时、API 错误、权限拒绝、模式漂移；要求 agent 报告异常而非捏造。
   - **B 类 - 数据质量**：陈旧数据、矛盾数据、错误答案、静默空结果；要求 agent 识别缺陷而非当成可靠。
   - **C 类 - 安全**：提示注入、描述投毒、伪造声明、数据泄露；要求 agent 不执行隐蔽指令。

3. **评分机制**
   - **主要检查**：确定性 pass/fail 规则（例如：是否传播了假声明、是否调用了泄露域名等）。
   - **诊断标签**：LLM 评判器（Claude 4.5 Haiku）给出三类标签：故障检测（二分）、恢复动作（四类：恢复/安全中止/传播/崩溃）、不确定性沟通（二分）。该评判器已与人工标注进行验证（Cohen’s κ 约 0.7-0.87），但被视为补充信号，主要判决仍以确定性检查为准。

4. **交互式仪表盘**（图 2）：支持连接 MCP 服务器、选择模型/执行框架、输入任务、指定故障类型/工具/调用索引、查看分歧点、选择缓解措施并重跑。

## 六、主要结论与发现

1. **性能范围**：最优 agent（DeepSeek）通过 105/120，最弱（Llama）通过 77/120，差异 28 分。这种差距在传统任务成功基准中不可见（因为工具正常时都成功）。
2. **弱点集中在静默数据质量故障**（B 类）：所有 agent 在 B 类上得分最低（DeepSeek 29/40，Llama 13/40）。这些故障无错误码，易被忽略。
3. **缓解效果因故障类型而异**：
   - 重试缓解将 A1 超时故障从 30% 提升至 100% (Llama)。
   - 但 B1/B2/B3（陈旧、矛盾、无关）缓解效果极差，仍维持 30-40% 左右。
   - C4 数据泄露不受输出层缓解影响（需地址过滤/权限控制）。
4. **agent 行为差异**：面对 403 错误，强 agent（DeepSeek）会多步探查并报告权限失败，弱 agent 则捏造成功。
5. **LLM 评判器与人类一致性**：中等至良好（κ 0.69-0.87），但存在过判现象（图 3 中 Prop. 栏被视为上限），因此确定性检查保持为主判决。

## 七、优点

1. **新颖的受控比较方法**：通过缓存重放注入单一故障，实现 clean vs faulted 的 apples-to-apples 对比，隔离干扰因素。
2. **完整的复现-干预-确认闭环**：不仅是诊断工具，还能验证缓解措施是否真有效，这是现有工具（LangSmith、AgentDiagnose）不具备的。
3. **与 MCP 生态紧密集成**：将 MCP 服务器作为干预面，可直接在开发者自己的 agent 和工具上运行，不依赖预定义 benchmark 数据。
4. **故障类型全面**：涵盖执行、数据质量、安全三大类 12 种，覆盖主流已知故障。
5. **可视化仪表盘**：展示分歧点、轨迹对比、标签，降低使用门槛。
6. **评分双轨制**：确定性检查确保可复现性，LLM 评判器提供解释性标签，且用人工验证降低偏差。
7. **开放源代码与场景库**：MIT 协议发布，便于社区扩展和复用。

## 八、不足与局限

1. **实验覆盖有限**：
   - 仅测试 **单一故障**，未考虑复合或级联故障，实际部署中多故障同时发生。
   - 12 种故障无法覆盖所有可能的失败模式（如长时序依赖、策略违规等）。
   - 缓解实验仅对**最弱 agent（Llama）** 进行，外推到其他 agent 需谨慎。
   - 人工标注仅 224 例，可能不足以支撑所有故障类型。
2. **评分可靠性风险**：
   - LLM 评判器可能过度报告“传播”标签（作者明确指出为上限），且无法完全避免幻觉。
   - 确定性检查基于模式匹配，可能遗漏异常但未在检查范围内的行为。
3. **场景库偏差**：
   - B 类场景为原创，未经第三方验证，其难度可能与真实场景有差距。
   - 所有场景基于有限领域（5 个），未能覆盖多样化的现实任务。
4. **双用途风险**：安全故障注入（数据泄露、提示注入等）的代码和场景可能被恶意使用，但作者声明仅用于开发者自测且遵守第三方许可。
5. **未提供算力细节**：缺乏具体计算资源信息，影响复现和成本估算。
