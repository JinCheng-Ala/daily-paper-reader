## 摘要
使用工具的LLM智能体大多在假设所有工具正常工作的条件下进行评估。当工具超时、返回过时一周的值，或在部署过程中其描述被污染时，开发人员需要一种可控的方式来复现故障、测试修复，并在部署前确认修复有效。我们提出AgentCheck，一个开源Web工作台，可将MCP服务器转变为干预界面。AgentCheck让智能体操作其真实工具并记录每次工具响应，然后通过故障注入器（12种类型）扰动响应后重新运行智能体。缓存中匹配的工具调用会被重放，而在智能体行为出现分歧后的后续工具调用则实时执行。这形成了一个“复现-干预-确认”循环：开发人员切换缓解措施，针对相同故障重新运行，观察故障是否消失。评分分为两部分：确定性通过/失败规则，以及用于解释性标签的LLM裁判（经人工标注验证）。在五个智能体上，最佳智能体在120个场景中通过了105个，最弱的仅通过77个。故障通常是静默的，即智能体自信地使用了不正确的工具输出而非崩溃。对于最弱的智能体，重试缓解措施将超时错误的成功率从低至30%提高到100%，而陈旧数据故障无论采用何种缓解措施，成功率仍保持在每10次约3-4次。AgentCheck使这些故障模式在部署前可复现、可比较且可验证。

## Abstract
Tool-using LLM agents are mostly evaluated assuming all tools work. When a tool times out, returns a week-stale value, or has its description poisoned in deployment, the developer needs a controlled way to reproduce the failure, test a fix, and confirm the fix worked before deployment. We present AgentCheck, an open-source web workbench that turns an MCP server into an intervention surface. AgentCheck runs an agent against its real tools and records every tool response, then re-runs the agent with the response perturbed by a fault (12 types) injector. Matching tool calls are replayed from cache, and later tool calls go live after the agent diverges. This yields a reproduce-intervene-confirm loop: the developer toggles a mitigation, re-runs against the identical fault, and sees if the failure goes away. Scoring has two parts: deterministic pass/fail rules, plus an LLM judge for interpretive labels, validated against human annotations. Across five agents, the best passes 105/120 scenarios and the weakest only 77. The failures are usually silent, confident use of incorrect tool outputs rather than crashes. On the weakest agent, a retry mitigation raises success on timeout error faults from as few as 30% of cases to 100%, whereas stale-data faults remain near 3-4 of 10 regardless of the mitigation. AgentCheck makes these failure modes reproducible, comparable, and verifiable before deployment.

## 2. 论文提出的方法论：核心思想、关键技术细节、算法流程

### 核心思想
- 将**MCP服务器**（Model Context Protocol Server）转变为**干预表面**（intervention surface），通过可控的故障注入和响应重放，实现对智能体行为的精确比较。

### 关键技术细节
1.  **可控比较（Controlled Comparison）**：
    - **干净运行（Clean Run）**：智能体面对真实工具，记录所有工具响应。
    - **故障运行（Faulted Run）**：在选定的注入点（p）使用故障注入器（g）扰动一个工具响应，其余响应从缓存重放。
    - **缓解后运行（Mitigated Run）**：应用缓解措施（m）后，针对相同的故障（p, g）重新运行，验证修复是否有效。
2.  **故障注入引擎（Fault Injection Engine）**：
    - 支持**12种故障类型**，分为三大类别：
        - **A类：工具执行故障**（超时、API错误、权限拒绝、模式漂移）
        - **B类：数据质量故障**（陈旧数据、矛盾数据、错误答案、空结果）
        - **C类：安全故障**（提示注入、描述投毒、虚假声明、数据外泄）
3.  **评分系统（Scoring）**：
    - **初级通过/失败检查（Primary Pass/Fail Checks）**：基于确定性规则，检查智能体在故障后的具体行为（如是否传播虚假声明、执行注入指令）。
    - **诊断标签（Diagnostic Labels）**：使用LLM裁判（LLM Judge）生成解释性标签（如是否检测到故障、是否恢复、是否传播、是否表达不确定性），经人工标注验证。
4.  **复现-干预-确认循环（Reproduce-Intervene-Confirm Loop）**：
    - 开发者复现故障 → 应用缓解措施 → 针对相同故障重新运行 → 确认故障是否消失。

### 算法流程（文字描述）
1.  **缓存构建**：在干净运行中，智能体与真实工具交互，记录每个工具调用的响应，构建缓存K。
2.  **故障注入与重放**：在故障运行中，重放缓存K，但在指定的注入点使用故障注入器g替换原始响应。
3.  **缓解验证**：在缓解运行中，应用缓解措施m（如重试、模式检查、注入过滤），再次使用相同的故障注入，若所有先前失败的检查均通过，则确认修复。
4.  **输出**：输出轨迹的分歧点、失败检查列表和修复确认标记。

1.  **智能体鲁棒性差异显著**：最佳智能体（DeepSeek）通过105/120个场景，最弱（Llama）仅通过77/120个，差距28分。这种差异在传统基准（假设工具正常）中无法体现。

3.  **缓解效果因故障类型而异**：
    - **重试**对工具执行故障（A1、A2、A3）效果显著（如Llama的超时故障成功率从30%提升至100%）。

4.  **成功示例**：DeepSeek在权限拒绝故障中能进行多步调查后报告失败，而较弱的智能体可能直接编造成功。
5.  **修复确认可行**：AgentCheck支持完整的复现-干预-确认循环，开发者可针对特定故障验证修复措施。
