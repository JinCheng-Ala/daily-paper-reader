## 摘要
智能体AI系统日益通过异构运行时环境行动：本地编码钩子、SDK工具、浏览器自动化、托管智能体轨迹、API网关和工作流引擎。因此，一个单一的操作行为，如发布代码、更改身份状态、转移资金或导出数据，可能由许多不兼容的运行时记录表示。这使得一个基本的治理问题难以回答：实际批准了什么行动？什么证据将批准与执行绑定？独立的验证者能否在之后重现相同的行动身份？本文提出了规范动作验证与证明（CAVA），这是一个运行时语义层，用于将异构智能体活动转换为规范的运行时动作对象。CAVA位于证据携带智能体行动（PCAA）之下：PCAA定义了部署者拥有的路线-审查-证明治理过程，而CAVA定义了该过程治理的稳定动作对象。本文形式化了规范动作身份、语义模式检测、批准绑定、收据完整性、运行时可移植投影以及可选的证明基底。我们通过一个包含96个种子、384个变体的基准测试研究了参考实现，涵盖语义等价性、语义分离、封装绕过、假阳性控制、批准绑定、收据可再现性、证明篡改检测、运行时可移植性、语义模式检测、策略退化以及Azure部署演练。本文的贡献在于将动作级规范化和策略可寻址语义模式作为部署者侧AI治理的必要基底进行系统化表述。

## Abstract
Agentic AI systems increasingly act through heterogeneous runtimes: local coding hooks, SDK tools, browser automation, managed-agent traces, API gateways, and workflow engines. A single operational act such as publishing code, changing identity state, moving money, or exporting data may therefore be represented by many incompatible runtime records. This makes a basic governance question difficult to answer: what action was actually approved, what evidence binds the approval to execution, and can an independent verifier reproduce the same action identity later?   This paper presents Canonical Action Verification and Attestation (CAVA), a runtime-semantics layer for converting heterogeneous agent activity into canonical runtime action objects. CAVA is positioned below Proof-Carrying Agent Actions (PCAA): PCAA defines the deployer-owned route-review-prove governance process, while CAVA defines the stable action object that process governs. The paper formalizes canonical action identity, semantic pattern detection, approval binding, receipt integrity, runtime-portable projection, and optional attestation substrates. We study a reference implementation through a 96-seed, 384-variant benchmark covering semantic equivalence, semantic separation, wrapper bypass, false-positive control, approval binding, receipt reproducibility, attestation tamper detection, runtime portability, semantic pattern detection, policy degradation, and Azure deployment drills. The contribution is a systems formulation of action-level canonicalization and policy-addressable semantic patterns as a necessary substrate for deployer-side AI governance.

### 论文提出的方法论

- **核心思想**：提出**规范动作验证与证明（CAVA）**层，它将异构的原始运行时事件转换为**规范运行时动作对象**（定义1），并计算**规范指纹**（定义2）。该指纹是防碰撞哈希，策略决策、批准收据和审计证据均绑定到指纹而非原始文本。
- **关键技术细节**：
    1. **规范动作对象**：包含 schema_version, runtime, adapter, executable, operation, category, systems_touched, reversible, target/subject 等字段。

    3. **语义模式层**：将规范动作、边界上下文和数据上下文映射为策略可寻址的模式（如公共持久性出口、安全控制弱化、委托权限不匹配等），避免成为客户特定异常集合。
    4. **与PCAA的关系**：PCAA定义审查-批准-证明治理循环，CAVA提供该循环治理的稳定动作对象。两者互补。
    5. **证明基底**：支持本地哈希收据、签名、可验证凭证、供应链证明、账本锚定等，但账本可选，非定义。

### 论文的主要结论与发现

- 原始文本治理没有稳定的语义等价概念；第一令牌规则无法处理封装且容易误报；规范收据允许批准和证明检查，原始字符串无法提供。
- 在代表语料中，CAVA在语义等价、封装绕过、批准绑定、收据重现、篡改检测、运行时可移植性、语义模式检测等所有指标上达到**1.0**，而基线在大多数任务上为0.0。
- CAVA为部署者侧AI治理提供了必要基底：动作级规范化和策略可寻址语义模式，使治理可测量而非空谈。

### 优点

- **系统化方法**：从问题形式化、协议设计、威胁模型到开源核心+商业附加层的明确边界，逻辑完整。
- **可验证性**：公共规范、确定性哈希、开放收据验证器，使第三方可独立复现核心语义和收据完整性。
- **基准设计**：定义清晰、指标对应失败模式，包含消融和红队案例，有助于评估和持续改进。
- **诚实披露**：明确承认语料代表性有限、解析器覆盖不全、观察模式不可等同于阻断等局限，避免过度宣传。
- **与PCAA等框架兼容**：定位清晰，不试图替代，而是作为治理栈的下层。

### 不足与局限

- **实验覆盖有限**：96种子语料远不能代表企业真实操作的分布。作者承认需要第三方痕迹和对抗性提交。
- **解析器覆盖依赖**：规范化的强度取决于适配器覆盖。私有系统或新运行时可能需要自定义解析器，解析失败可能导致漏报或误报。
- **披露边界**：生产解析包和企业策略阈值被隐藏，限制了完全复现管理行为的能力。
- **基线较弱**：仅与原始文本和第一令牌对比，未与基于LLM的语义分析或更高级的规范形式对比，可能高估CAVA的优势。
- **证明范围有限**：收据哈希和签名证明完整性，但不保证业务决策质量。
- **运行时深度差异**：某些运行时仅能观察，不能内联阻断。CAVA必须诚实地披露该差异，但论文强调这一点，属于有意约束而非忽略。
- **商业依赖**：开源核心是骨架，完整能力需要商业产品，可能产生供应商锁定风险（尽管作者通过开放接口减轻）。
