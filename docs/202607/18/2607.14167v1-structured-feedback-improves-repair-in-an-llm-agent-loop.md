## 摘要
LLM智能体通常在外部验证拒绝候选方案后重试，但验证与下一个模型调用之间的接口仍未明确。我们引入了VeriHarness，一个代码控制的智能体循环，其中模型生成候选方案，而外部验证者控制接受、预算和追踪。我们利用它来比较原始诊断与能够识别失败位置、观测值和可接受替代方案的反馈。在四轮调用限制下的50个配对TextWorld游戏中，包含所有三个字段的反馈将Qwen2.5-Coder-14B的终端成功率从14/50提升到36/50（+44个百分点），Llama-3.1-8B从8/50提升到29/50（+42个百分点）。消融实验发现主要增益来自可接受替代方案：仅包含位置和观测值的反馈仍接近原始诊断基线。以散文形式而非键值JSON记录呈现完整的修复信息取得了几乎相同的成功率，没有证据表明JSON语法本身改善了修复。这种顺序在测试的调用预算和一种采样解码设置中保持一致。

## Abstract
LLM agents often retry after external validation rejects a candidate, but the interface between validation and the next model call remains underspecified. We introduce VeriHarness, a code-controlled agent loop in which models generate candidates while external validators control acceptance, budgets, and traces. We use it to compare raw diagnostics with feedback that identifies the failure location, observed value, and admissible alternatives. Across 50 paired TextWorld games under a four-call cap, feedback containing all three fields raises terminal success from 14/50 to 36/50 for Qwen2.5-Coder-14B (+44 percentage points) and from 8/50 to 29/50 for Llama-3.1-8B (+42 points). Ablations locate most of the gain in the admissible alternatives: feedback containing only the location and observed value remains near the raw diagnostic baseline. Presenting the complete repair information in prose instead of a keyed JSON record yields nearly the same success, providing no evidence that JSON syntax itself improves repair. The ordering persists across the tested call budgets and one sampled-decoding setting.

## 1. 论文的核心问题与整体含义（研究动机和背景）
LLM 智能体在外部验证拒绝候选方案后通常会重试，但验证与下一个模型调用之间的接口往往过于粗糙——仅返回一条原始拒绝消息（如“命令不可行”），缺乏修复所需的具体信息。本文旨在回答：**验证器应该向模型返回什么样的反馈才能最有效地提升修复成功率？** 具体地，研究了包含“失败位置”“观测值”和“可接受替代方案”的结构化反馈 vs. 原始诊断的效果，并进一步分析各字段的贡献以及表示形式（JSON 键值 vs. 自然语言）的影响。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：提出 **VeriHarness**，一个代码控制的智能体循环，将生成（LLM worker）与验证（外部 gate）严格分离。代码负责维护持久状态、执行预算、确定性验证；模型仅接收紧凑的任务状态和当前修复反馈。验证器返回的失败信息通过一个“失败编码器”映射到统一接口（标签、位置、观测值、可接受替代方案）。
- **关键技术细节**：定义四种反馈策略：
  - **RawDiag**：原始验证错误消息（无显式修复字段）。
  - **SameNL**：以自然语言形式给出位置、观测值和可接受的替代方案。
  - **LocObs**：JSON 键值格式，包含标签、位置和观测值，但**省略**可接受替代方案。
  - **TypedFields**：JSON 键值格式，包含标签、位置、观测值和可接受替代方案（与 SameNL 信息相同但格式不同）。
- 通过对比这些策略，可分别检验：① 是否包含替代方案（LocObs vs. TypedFields）、② 表示形式（SameNL vs. TypedFields）、③ 结构化信息整体（RawDiag vs. SameNL）对修复的影响。

## 6. 论文的主要结论与发现
- **RQ1**：包含失败位置、观测值和可接受替代方案的结构化反馈显著提升了终端成功率：
  - Qwen2.5-Coder-14B：28% → 72%（+44 百分点，95% CI [28,60]）。
  - Llama-3.1-8B：16% → 58%（+42 百分点，95% CI [28,56]）。
- **RQ2**：增益主要来源于**可接受替代方案**字段：
  - LocObs（仅位置+观测值）与 RawDiag 几乎无差异。
  - SameNL 与 TypedFields（都包含替代方案）成功率几乎相同（差距 ≤2 百分点），表明 JSON 键值格式本身不带来额外优势。
- **预算维度**：结构化反馈在增加调用次数时持续改善，而原始诊断在 B=4 后停滞。
- **HumanEval 范围检查**：当验证器无法暴露失败（如仅测试一个公开用例，而隐藏用例出错）时，任何反馈策略都无法修复，说明**反馈有效的前提是验证器能捕捉到失败**。

## 7. 优点
- **控制良好**：通过代码控制循环，确保反馈策略是唯一变量，避免模型自行管理状态带来的干扰。
- **消融设计清晰**：通过 LocObs 和 SameNL/ TypedFields 拆解了“位置+观测值”和“可接受替代方案”的贡献，并对比了自然语言与 JSON 格式。
- **统计严谨**：使用配对 bootstrap 置信区间和 Holm 校正的 McNemar 检验，避免多重比较膨胀。
- **实验范围多样**：包含预算灵敏度、采样解码、HumanEval 范围检查，提升了结论的鲁棒性。
- **可复现**：开源了 VeriHarness 源码、全部 traces 及分析脚本。

## 8. 不足与局限
- **场景单一**：主要实验基于 TextWorld 游戏，与现实软件修复（如代码测试、CI 流程）仍有差距，可接受替代方案的枚举在真实场景中往往不可能。
- **模型与硬件有限**：仅测试了两种量化模型（14B、8B），均运行于单卡 L4/T4，未覆盖更大模型或未量化版本；HumanEval 仅 15 个任务，样本量过小。
- **可接受替代方案列表固定**：取环境顺序的前 12 条，未测试更长、排序或随机顺序对结果的影响。
- **表示对比不完全纯净**：SameNL（散文）与 TypedFields（JSON）除了格式外，前者缺少“失败标签”，且两者框架文本不同，不是纯标点对比。
- **策略名影响首次调用**：首次调用上下文包含策略名（例如“RawDiag” vs “TypedFields”），导致首次成功率有微小差异，无法严格保证首次提示完全一致。
- **验证器不完整问题**：HumanEval 实验表明反馈无法修复验证器未暴露的失败，这在实际应用中非常普遍（如测试覆盖率不足），论文仅将其作为范围检查，未提出解决方案。
