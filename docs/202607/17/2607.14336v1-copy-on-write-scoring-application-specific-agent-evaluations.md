## 摘要
在软件系统中可信地部署基于LLM的智能体，需要评估它们在特定应用工作流上的表现，并具备足够的粒度以定位其成功和失败之处。然而，现有的智能体评估机制存在局限性：基准测试对特定应用工作流和环境的构建效度较低，而复制评估环境成本高昂且容易发生漂移。我们提出写时复制（CoW）评分，这是一个框架，通过使用PostgreSQL级别的写时复制机制直接在应用环境中评估智能体操作，以隔离智能体的写入操作。CoW评分生成会话级别和操作级别的分数，突出显示智能体在特定应用环境中的数据库写入操作成功与失败的位置，从而实现对智能体工具和接口的低成本评估和迭代。我们在开源项目管理平台Plane上演示了该框架，分析揭示了工具接口中的特定问题，相应的修复对受影响的模型产生了可衡量的改进。Python库：https://github.com/trail-ml/agent-cow-python

## Abstract
Trustworthy deployment of LLM-based agents in software systems requires evaluating how they perform on application-specific workflows, with enough granularity to localize where they succeed and fail. Yet existing agent evaluation mechanisms are limited: benchmarks have low construct validity for application-specific workflows and environments, and replica evaluation environments are expensive and prone to drift. We propose Copy-on-Write (CoW) Scoring, a framework that evaluates agent operations directly within application environments using a PostgreSQL-level Copy-on-Write mechanism to isolate agent writes. CoW Scoring produces session- and operation-level scores that highlight where agents' database write operations succeed and fail in a given application environment, enabling inexpensive evaluation and iteration on agent harnesses and tool surfaces. We demonstrate the framework on Plane, an open-source project-management platform, where analysis surfaced specific issues in the tool surface, and corresponding fixes produced measurable improvements on affected models. - **整体含义**：提出一种新的评估范式——写时复制评分（Copy-on-Write Scoring），通过数据库层面的写隔离机制，在真实生产级平台上安全、细粒度地评估agent的写操作，支持“评分→诊断→修复”的闭环迭代，提升agent部署的可靠性和安全性。

## 2. 方法论：核心思想、关键技术细节、公式/算法流程

### 2.1 核心思想
- **写时复制（CoW）机制**：借鉴操作系统中的写时复制概念，应用于数据库。agent的所有写操作不直接修改生产数据，而是被截获并写入一个**独立的变更表（changes table）**，通过视图（view）合并基础表（base table）和变更表，实现读写隔离。
- **评分框架**：将agent会话（session）的写操作与人工记录的标准答案（ground-truth, GT）会话进行比较，在**结构**（哪些表、行、列被修改）和**内容**（具体写入的值）两个维度上，计算**会话级**和**操作级**分数。

### 2.2 关键技术细节
- **CoW机制实现**（基于PostgreSQL）：
  1. 为每个原表（如`tasks`）创建基础表`tasks_base`（存储生产数据）和变更表`tasks_changes`（额外包含`session_id`, `operation_id`, `cow_updated_at`, `cow_deleted`元数据列）。
  2. 创建同名视图（如`tasks`），通过合并查询`tasks_base + tasks_changes`展示当前会话可见的数据。
  3. 在视图上定义`INSTEAD OF`触发器，将所有写操作重定向到`tasks_changes`表。
  4. agent操作时需在请求头中传递`session_id`和`operation_id`。
- **CoW集成工作量**：以Plane为例，CoW核心逻辑约250行SQL，记录基础设施约540行（可移植），总约790行。

### 2.3 评分公式与算法流程
- **会话级评分**：
  - **结构得分** \( s_{struct} = \frac{N_{matched}}{N_{matched} + N_{missing} + N_{extra}} \)
    - \( N_{matched} \)：GT和agent都修改且匹配的行数（通过UUID映射）
    - \( N_{missing} \)：GT中有但agent未产生的行
    - \( N_{extra} \)：agent额外产生的行
  - **内容得分** \( s_{content} = \frac{1}{N_{matched}} \sum_{(g,a) \in matched} \text{sim}(g,a) \)
    - 每个字段按数据类型默认比较器计算相似度（文本用`SequenceMatcher.ratio()`，JSON用深度相等，其他精确相等），用户可自定义。
  - **整体得分** \( s_{overall} = 0.5 \cdot s_{struct} + 0.5 \cdot s_{content} \)
- **操作级评分**（评估每个操作的贡献）：
  - **结构效用** \( u_{struct}(o_i) = s_{struct}(G_{\leq i}) - s_{struct}(G_{< i}) \)：应用操作\(o_i\)前后结构得分的增量。
  - **内容效用** \( u_{content}(o_i) = \frac{1}{|M_i|} \sum_{(g,a) \in M_i} \text{sim}(g,a) \)：操作\(o_i\)所写行中匹配行的平均相似度。
- **评分流水线**（五个阶段）：
  1. 提取：从变更表读取行，按`operation_id`分组，按时间排序。
  2. 匹配：每张表按主键取最后写入状态，通过贪婪算法找到GT和agent实体间的最佳映射（UUID桥接不同主键）。
  3. 字段比较：按SQL类型比较每个字段。
  4. 逐操作评分：按拓扑顺序累计agent行，与GT重新打分，记录增量。
  5. 归约：应用注册的评分函数（如整体、精确率、召回率、F1）输出最终标量。

## 6. 主要结论与发现

1. **CoW机制可行且低成本**：在PostgreSQL应用中集成CoW只需约250行核心代码，工具表面改进后模型性能显著提升。
2. **分数有效反映性能排序**：模型得分顺序与公开基准一致（如Gemini-3.1-Pro > Gemini-3.1-Flash-Lite，GPT-5 > GPT-4.1），前两轮平均得分标准差小（≤0.03），表明结果稳定。
3. **诊断出具体失败模式**：
   - **词汇不匹配**：Plane API使用“work item”而提示词常见“issue”，导致Gemini-2.5-Pro等模型放弃搜索或使用错误字符串，属于信息检索中著名的“词汇问题”。
   - **幻觉与额外写入**：GPT-4.1有时在未调用`discover`前猜测不存在的操作名，导致API错误（良性）；Gemini-3.1-Flash-Lite产生大量未提示的额外写入（如47个标签和分配者），在传统评估中极易被忽略。
4. **工具表面改进带来显著提升**：在API描述中补充“issues”后，受词汇不匹配影响最大的模型提升最大（Gemini-2.5-Pro提高54%，GPT-4.1提高47%），其他模型也有小幅改善。
5. **操作级分数提供更细粒度诊断**：可以定位具体哪些操作有益（增量正）或有害（增量负），有利于优化提示或工具设计。

## 7. 优点

- **高构建效度**：直接在真实应用环境中评估，避免副本漂移，评估结果更贴近实际部署表现。
- **低成本与可迭代**：无需搭建重复环境，工具表面修改后可快速重新评估，支持闭环改进。
- **细粒度分数**：会话级和操作级分数分别从宏观和微观角度揭示成功与失败，操作级效用函数可赋予部分成功以正信号，适合强化学习（如GRPO）或遗传编程（如GEPA）优化。
- **安全隔离**：CoW机制本身可作为运行时安全保障，防止意外写入生产数据。
- **可定制性**：用户可自定义字段比较器和评分函数，适配不同业务需求。
- **开源实现**：提供了Python库`agent-cow`，降低采用门槛。

## 8. 不足与局限

1. **GT会话需手动创建**：目前需要人工或LLM辅助生成并审核GT工作流，耗时且可能产生偏差，限制了规模化应用。未来可探索自动生成（如MCTS+LLM）后人工审计。
2. **假设提示词完全指定预期结果**：真实用户常使用模糊提示，多个合理终态会被判为失败。可通过自定义比较器、词汇多样性、分离结构和内容得分部分缓解。
3. **仅覆盖写操作**：agent的读操作（如冗余`discover`调用）未被评分，而论文实验显示这些调用本身携带诊断信号（如词汇不匹配）。未来可扩展读信号分析。
4. **限于PostgreSQL**：当前实现依赖PostgreSQL的视图和触发器机制，扩展到其他数据库（如MySQL、MongoDB）需要额外工作。
5. **实验规模有限**：仅一个应用（Plane）、20个工作流、5个模型、少量重复，统计分析说服力不足。更多应用和更大规模实验才能验证框架普适性。
6. **可能给予假信心**：若GT工作流不具代表性，高分可能掩盖其他错误行为。建议搭配性能监控和异常检测。
7. **处理复杂触发器不足**：论文承认对级联触发器和PostgreSQL特定触发器的处理需进一步研究，以增强实际可移植性。
