## 摘要
智能体技能封装了供通用智能体使用的程序化指令和检查，但加载技能并不能保证每个要求在长时间的工具使用轨迹中始终保持激活状态。我们在一个源自生产环境的白盒代码审计工作流中研究此问题。在保持任务和24个工件检查不变的情况下，我们变化周围上下文，并对失败首次出现的环节进行分类：需求丢失、编辑漂移、检查失败，或非智能体检器/运行时失败。使用gpt-5.4-mini的Codex在10,991字符的简洁上下文中通过10次运行中的8次，但在299,140字符的相关上下文和等长的无关上下文中仅通过3次。这50个百分点的差异很大，但在双侧Fisher检验下仍处于趋势水平（p=0.0698）。然而，在两种长上下文条件下，需求覆盖率仍保持在92%以上，表明少数遗漏可能使原本完整的工件无效。第二个任务在所有简洁和长上下文运行中均通过，因此证据不支持存在通用的上下文长度阈值。使用详细的外部检查表在10次运行中通过10次，而通用自我检查通过5次（p=0.0325）。编码智能体框架可能通过选择更小的工作集来提供帮助，但并不能消除失败。我们不引入上下文腐烂或新的通用监控方法；我们为白盒代码审计提供有界的失败分类和实证案例研究。

## Abstract
Agent Skills package procedural instructions and checks for use by general-purpose agents, but loading a skill does not guarantee that every requirement remains active throughout a long tool-using trajectory. We study this problem in a production-derived, white-box code-audit workflow. Holding the task and 24 artifact checks fixed, we vary the surrounding context and classify where failures first become visible: lost requirements, editing drift, failed checking, or non-agent evaluator/runtime failures. Codex with gpt-5.4-mini passes 8/10 runs in a 10,991-character clean context but only 3/10 in both a 299,140-character relevant context and an equal-length irrelevant context. This 50-percentage-point difference is large but remains trend-level under two-sided Fisher tests (p = 0.0698). Requirement coverage nevertheless stays above 92% in both long conditions, showing that a few omissions can invalidate an otherwise complete artifact. A second task passes all clean and long runs, so the evidence does not support a universal context-length threshold. A detailed external checklist passes 10/10 runs, compared with 5/10 for a generic self-check (p = 0.0325). Coding-agent scaffolds may help by selecting a smaller working set, but they do not eliminate failures. We do not introduce context rot or a new general monitoring method; we provide a bounded failure classification and empirical case study for white-box code auditing.

- **研究动机**：编码智能体依赖可复用的“技能”包（程序化指令、引用和检查），但加载技能并不能保证所有要求在整个工具使用轨迹中始终保持激活。尤其在代码审计场景中，一个缺失的字段或一次越权编辑就可能导致整个工件无效。现有长上下文研究主要关注模型可用窗口退化，而智能体指令增加了结构复杂性和约束跟踪负担，但尚缺乏在固定技能任务下系统观察哪些要求失败以及为何失败的白盒实证。
- **核心问题**：当基于技能的固定任务被置于不同工作上下文（简洁、长相关、长无关）时，哪些显式要求会失败？一个简单的外部检查清单能否防止这些失败？
- **整体含义**：论文不是提出新的理论框架或监控语言，而是提供一个透明的案例研究、一个可复用的白盒失败分类程序，以及关于简单外部检查何时有效的初步证据，为更大规模的预注册研究奠定基础。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **白盒设置**：研究在“白盒”代码审计工作流中进行——评估者可以观察到技能指令、智能体读取的文件、编辑、工具日志、完成消息和外部评分后的工件。关键要求（如“所有复数证据字段必须是数组”、“不得更改受保护的核心字段”）被提取为24个可观察检查（确定性断言），并经过正负向夹具验证。
- **上下文构建**：保持任务、起始工件、工具和24个检查固定，仅改变周围上下文。定义了五个主要提示条件：
  - Clean（10,991字符）：最小完整任务上下文
  - Relevant long（299,140字符）：添加同一工作流的生产材料
  - Irrelevant long（299,140字符）：等长的无关自然档案文字
  - Generic check（299,433字符）：在Relevant long基础上加入通用自检指令
  - Detailed check（300,082字符）：在Relevant long基础上显式列出所有24个检查项
- **失败分类法**：对于每个失败运行，根据最早可见证据将其归为四类之一：
  1. **Lost requirement（需求丢失）**：强制性条目从智能体的任务表征或最终工件中缺失，无活跃保留证据。
  2. **Editing drift（编辑漂移）**：要求存在但编辑违反了它（如覆盖受保护字段、替换邻近问题）。
  3. **Failed checking（检查失败）**：工件或工具输出暴露错误，但验证步骤未能重新打开或纠正。
  4. **Non-agent failure（非智能体失败）**：由评估者或运行时导致（如语义假阴性、流断开）。
- **统计方法**：报告严格任务成功率、要求覆盖率、95% Wilson区间；主要2×2比较使用双侧Fisher精确检验（α=0.05）。运行间检查非独立，因此不将240个检查作为独立样本。

- **工件检查**：24个检查覆盖：必填的复数字段存在且为数组、字段基数上限、任务特定证据与判别器、输出模式类型、受保护字段相等、最终工件可解析且完整。
- **对比条件**：
  - 上下文长度：Clean vs. Relevant long vs. Irrelevant long
  - 检查清单干预：Generic vs. Detailed（在Relevant long下）

## 6. 论文的主要结论与发现

1. **长上下文降低主任务通过率**：洁净上下文通过8/10，两个299K长上下文均通过3/10。观察到的失败率从20%增至70%，但Fisher检验p=0.0698，统计不显著。
2. **语义相关性不区分效果**：相关与无关长上下文二进制结果相同（3/10），Fisher p=1.0。排除“相同领域材料更有害”的简单假设。
3. **失败形式稀疏但致命**：长上下文下载体要求覆盖率仍>92%（相关92.1%，无关93.8%），但任务成功率仅30%。一个缺失数组（如T002的`state_relation_lenses`）即可使完美工件无效。
4. **详细外部检查清单有效**：Detailed pass 10/10，Generic pass 5/10（p=0.0325）。机制推测：Generic要求智能体重建验证集，可能继承生成中的遗漏；Detailed提供独立要求来源。
5. **框架选择可部分缓解**：对gpt-5.5，Bundled Direct（小证据包）和Codex均通过3/3，而Full Direct仅1/3，说明选择更小工作集比单纯使用工具更重要。但对gpt-5.4-mini，Codex在洁净下辅助但有，在长上下文中仍不稳定。
6. **第二任务稳定反例**：Revert SL-007所有8次运行（2清洁+3相关+3无关）均通过，证明长上下文效应非普遍，取决于具体任务-模型对。
7. **基础设施可靠性问题**：n=10扩展中仅50.9%的尝试产生可评分输出（57次尝试29个有效），Irrelevant条件最差（25次尝试仅5个可评分，20次因首次token断开等失败），表明评估系统本身是独立失效源。

## 7. 优点

- **白盒设计**：研究者可以观测指令、工件、工具日志和外部检查结果，能够明确地将失败定位到具体可见要求，而非依赖模拟内部推理。
- **细粒度失败分类**：四类失败（需求丢失、编辑漂移、检查失败、非智能体失败）提供了比“任务通过/失败”二元标签更丰富的干预信息，且分类规则有序、可复现。
- **控制变量严谨**：固定任务、起始工件、检查器，只改变上下文，避免答案泄露（通过清理历史文件），并明确区分有效运行与基础设施失败。
- **跨任务/跨模型边界探针**：不仅报告主效应，还通过稳定反例和不同模型的表现界定了结论范围，避免过度泛化。
- **使用外部检查清单**：提供了低成本、可落地的缓解方案，并且解释了其相对于通用自检的优势在于“恢复遗漏的要求”，而不是依赖于智能体的内部记忆。
- **公开审计和成本核算**：附录包含详细失败编码指南、条件清单、中断尝试审计和完整成本台账，增强了可复现性。

## 8. 不足与局限

- **样本量小、统计不显著**：主效应p=0.0698，未达到α=0.05；Wilson区间宽，结果不稳定（如Irrelevant从0/5变为3/10）。作者承认这不等于确证性结论。
- **仅一个主要模型-任务对**：主要实验仅使用gpt-5.4-mini + Codex，且仅一个生产推导工作流；缺乏独立的高质量正复制（第二任务表现完美，无法提供效应）。
- **跨任务泛化严重不足**：四个候选任务因清洁能力门失败，无法用于上下文效应比较，唯一稳定反例无法提供效应估计支持。
- **基础设施缺失非随机**：28次中断尝试中20次集中在Irrelevant条件下，可能引入偏差（若易失败生成更易断开，则有效运行成绩可能被高估；若为随机网关事件，则低估）。
- **检查器覆盖不完全**：24个检查对字段存在、类型、基数、保护内容强，但对语义证据覆盖弱；通过检查的工件仍可能是浅层审计。
- **实验设计非预注册**：样本量扩展基于中间结果，统计测试的p值应视为描述性而非确证性。
- **无多变量因果分析**：上下文长度、语义相似性、活跃义务数量、证据位置等因素混杂，实验仅能检测关联而非分离因果。
- **框架变体不可比**：Direct、Codex、Claude Code的token预算不同，协议支持不同，因此不能排名框架，只能作为机制探针。
- **实际部署相关性**：研究场景为结构化代码审计，结果可能不适用于开放领域任务（如浏览器使用、文档编辑、自由对话）。
