## 摘要
LLM Agent的故障难以调试，因为错误出现的步骤往往不是导致错误的步骤。现有的可观测性工具重放执行轨迹，但在识别根本原因或将诊断转化为恢复方面支持有限。我们提出AgentDebugX，一个开源调试框架，将调试组织为检测、归因、恢复和重新运行的闭环。其核心DeepDebug通过全局轨迹理解、结构引导调查和交叉检验进行多轮根本原因诊断。在Who and When基准测试中，DeepDebug在所有评估方法中，在测试的两个开源主干网络上均实现了最佳严格归因准确率，在qwen3.5-9b上达到28.8%的精确Agent与步骤准确率，而最强的单次基线为21.7%。在GAIA上，DeepDebug在一次重新运行中修复了73个失败任务中的13个，而三个解耦的自校正基线修复了4到6个，将整体准确率从55.8%提升至63.6%。AgentDebugX通过Python库、命令行界面、Web控制台和可安装的Agent技能公开此工作流，并提供一个可选加入的错误中心，用于共享经过清洗的故障-诊断-修复包，并作为调试记忆重复使用。

## Abstract
LLM agent failures are difficult to debug because the step where an error surfaces is often not the one that caused it. Existing observability tools replay execution traces but provide little support for identifying the root cause or translating diagnosis into recovery. We present AgentDebugX, an open-source debugging framework that organizes debugging as a closed loop of Detect, Attribute, Recover, and Rerun. At its core, DeepDebug performs multi-turn root-cause diagnosis through global trajectory understanding, structure-guided investigation, and cross-examination. On the Who and When benchmark, DeepDebug achieves the best strict attribution accuracy among the evaluated methods on both tested open-weight backbones, reaching 28.8 percent exact agent-and-step accuracy on qwen3.5-9b versus 21.7 percent for the strongest single-pass baseline. On GAIA, DeepDebug repairs 13 of 73 failed tasks in a single rerun, compared with 4 to 6 for three decoupled self-correction baselines, improving overall accuracy from 55.8 percent to 63.6 percent. AgentDebugX exposes this workflow through a Python library, CLI, web console, and installable agentic skill, and provides an opt-in Error Hub for sharing scrubbed failure-diagnosis-repair bundles and reusing them as debugging memory.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 构建**闭环调试流程**：**Detect（检测）→ Attribute（归因）→ Recover（恢复）→ Rerun（重跑）**，并支持多轮迭代。
- 核心组件**DeepDebug**：一个多轮根因诊断Agent，通过**全局轨迹阅读→结构引导探查→交叉验证→诊断与建议**四阶段输出结构化报告（包括责任Agent、责任步骤、证据、解释、修复建议）。

### 关键技术细节
1. **轨迹捕获与统一表示**：将各种Agent框架（LangGraph、CrewAI、OpenAI Agents SDK、OpenTelemetry、ReAct等）的事件流转换为**AgentTrajectory**（由AgentEvent列表组成），独立于原始框架，支持离线重新分析和跨版本比较。
2. **Detect阶段**：
   - 确定性规则包（无模型调用）：检测格式错误、无限循环、提前成功等。
   - LLM Judge：读取目标与轨迹窗口，返回结构化发现（包括失败模式、置信度），共19种失败模式（规划、记忆、工具使用、验证、协调等）。
3. **Attribute阶段**：
   - 提供多种归因策略（成本-精度权衡）：启发式规则、单次全局阅读、步进检查、二分搜索、预算集成。
   - 复杂案例升级至DeepDebug。
4. **DeepDebug的多轮归因流程**（核心创新）：
   - **Stage 1 全局阅读**：读取整个轨迹，重构目标与历史，生成初始候选步骤。
   - **Stage 2 结构引导探查**：根据轨迹形状选择策略——多Agent场景沿级联链向上游回溯；单Agent场景二分法缩小范围，生成独立第二候选。
   - **Stage 3 交叉验证**：若两候选一致则接受；若不一致，侧边比对两个候选的上下文、输入输出和下游影响，选出因果解释更强的那个。
   - **Stage 4 诊断与建议**：输出结构化报告（责任Agent/步骤、解释、引注证据、一个具体修复建议）。
5. **Recover阶段**：
   - 原生路径直接使用DeepDebug的修复建议作为重试指令；也支持Reflexion、CRITIC、AutoManual等替代策略。
   - 所有修复建议需经人工或策略门控（human/policy gate）后才执行。
6. **Rerun阶段**：
   - 打包诊断、检查点、重试指令生成新的轨迹分支，并与原始轨迹并列比较。
   - 若重跑仍失败，重新进入Detect循环。
7. **扩展型失败分类法**：
   - 种子分类法包含19种模式；支持通过聚类候选新模式（归纳器）建议扩展，需人工批准。
8. **Error Hub**：
   - 可选共享的故障-诊断-修复捆绑包，经清洗（剥离敏感信息）后可作为CI回归测试、跨团队经验库，并支持DeepDebug的检索增强诊断。

## 7. 优点

- **完整的闭环设计**：将检测、归因、恢复、重跑集成于统一框架，且支持多轮迭代，解决现有工具仅停于观察的痛点。
- **框架无关的轨迹表示**：AgentTrajectory统一多种Agent框架，使得诊断逻辑和工具与具体运行时解耦，便于比较和复用。

- **可观测性标准**：框架无关的 AgentTrajectory 定义有望推动 Agent 轨迹标准化，便于后续研究在统一数据格式上进行根因分析。

## 12. 值得改进或进一步探讨的方向

- **自适应模型路由**：实验表明在强模型（如 gpt-5.4-mini）上单次阅读已足够，DeepDebug 的多次调用并不增益。未来可以设计基于轨迹复杂度预测的诊断开销-精度模型，动态选择归因策略。
- **扩展到多 Agent 系统**：目前的结构引导探查已支持多 Agent 回溯，但实验仅测试了简单的谁是谁（Who and When）基准，未在复杂多 Agent 协作场景（如 AutoGen 下的多轮辩论、信息传递网络）中验证。
- **修复建议的自动化验证**：当前修复建议需经人工或策略门控后才执行，未来可探索自动验证修复效果（例如基于差异对比的补丁验证），进一步提升闭环效率。
- **Error Hub 的检索增强诊断**：作者提及“检索增强诊断”但未评测。检索类似故障的已有修复方案可以加速归因，但如何保证检索结果的相关性与安全性（避免引入偏见或敏感信息）需要探索。
- **用户研究**：缺少开发者实际使用的时间效率、认知负荷、故障定位成功率的定量评估，建议后续开展受控实验。

## 13. 总结评价

AgentDebugX 提出并实现了一个完整的 Agent 调试闭环，以 DeepDebug 为核心实现了业界领先的归因精度，并在 GAIA 上展示了修复增益。其**框架无关性、成本-精度可调设计、Error Hub 共享机制**使其具有良好的可扩展性和实用性。尽管实验规模有限、部分对比公平性存在瑕疵、且对用户体验未做评估，但该工作足够新颖且工程化程度高，对 Agent 可观测性和自修复领域有重要推动作用。后续若能补充更多场景的评测、优化模型路由策略并加强社区生态建设，有望成为 Agent 开发的标准调试工具。
