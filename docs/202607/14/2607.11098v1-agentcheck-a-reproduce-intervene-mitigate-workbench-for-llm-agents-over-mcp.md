## 摘要
使用工具的LLM智能体通常在假设所有工具正常工作的条件下进行评估。当工具超时、返回过时值或在部署中其描述被污染时，开发人员需要一种可控的方式来复现故障、测试修复并在部署前确认修复有效。我们提出AgentCheck，一个开源Web工作台，它将MCP服务器转变为干预界面。AgentCheck让智能体在其真实工具上运行并记录每次工具响应，然后通过故障（12种类型）注入器的扰动响应重新运行智能体。匹配的工具调用从缓存中重放，而在智能体出现分歧后的后续工具调用则进行实时调用。这形成了一个复现-干预-确认循环：开发人员切换缓解措施，针对相同故障重新运行，并观察故障是否消失。评分分为两部分：确定性通过/失败规则，以及用于解释性标签的LLM评审员，并与人工标注进行验证。在五个智能体中，最佳智能体通过105/120个场景，最弱仅通过77个。失败通常是无声的、自信地使用错误的工具输出而非崩溃。在最弱智能体上，重试缓解措施将超时错误故障的成功率从低至30%提高到100%，而陈旧数据故障无论采取何种缓解措施仍保持在约10次中的3-4次。AgentCheck使这些故障模式在部署前可复现、可比较且可验证。

## Abstract
Tool-using LLM agents are mostly evaluated assuming all tools work. When a tool times out, returns a week-stale value, or has its description poisoned in deployment, the developer needs a controlled way to reproduce the failure, test a fix, and confirm the fix worked before deployment. We present AgentCheck, an open-source web workbench that turns an MCP server into an intervention surface. AgentCheck runs an agent against its real tools and records every tool response, then re-runs the agent with the response perturbed by a fault (12 types) injector. Matching tool calls are replayed from cache, and later tool calls go live after the agent diverges. This yields a reproduce-intervene-confirm loop: the developer toggles a mitigation, re-runs against the identical fault, and sees if the failure goes away. Scoring has two parts: deterministic pass/fail rules, plus an LLM judge for interpretive labels, validated against human annotations. Across five agents, the best passes 105/120 scenarios and the weakest only 77. The failures are usually silent, confident use of incorrect tool outputs rather than crashes. On the weakest agent, a retry mitigation raises success on timeout error faults from as few as 30% of cases to 100%, whereas stale-data faults remain near 3-4 of 10 regardless of the mitigation. AgentCheck makes these failure modes reproducible, comparable, and verifiable before deployment.

- **整体含义**：解决这一差距对于构建鲁棒、安全的 LLM 智能体至关重要。本文提出 AgentCheck，旨在将 MCP（模型上下文协议）服务器转变为干预表面，使开发人员能够在部署前系统性地测试智能体对工具故障的响应，并通过“复现-干预-确认”循环验证缓解措施的有效性。

## 2. 方法论

### 核心思想
- 将 MCP 服务器作为干预表面，通过**控制比较（controlled comparison）** 来隔离故障影响。
- 基本流程（Algorithm 1）：
  1. **清洁运行**：智能体对其真实工具执行任务，记录所有工具响应到缓存。
  2. **故障运行**：使用注入器 `g` 修改选定的一个工具响应（在注入点 `p`），其余响应从缓存重放；智能体执行直到完成或分歧。
  3. **缓解运行（可选）**：应用候选缓解 `m` 包装智能体，重新运行与故障运行完全相同的缓存和故障（同一 `(p, g)`），观察故障是否仍然触发。
- 通过这种方式，智能体行为差异唯一归因于注入的故障和施加的缓解。

### 关键技术细节
- **故障注入引擎**：拦截智能体与工具之间的通信，根据不同故障类型修改响应。共定义 12 种故障类型，分为三类：
  - **A类（工具执行）**：超时、API 错误、权限拒绝、模式漂移。
  - **B类（数据质量）**：陈旧数据、矛盾数据、错误回答、静默空结果。
  - **C类（安全）**：提示注入、描述投毒、虚假声明、数据外泄。
- **缓存重放**：故障运行和缓解运行中，匹配工具调用（工具名+参数+调用索引）直接从清洁运行缓存读取，只有注入点使用修改后的响应；后续未匹配的调用则实时执行。
- **评分机制**：两部分输出：
  - 主判断：确定性通过/失败检查，基于预定义规则（如未制造错误、未执行隐藏指令）。
  - 诊断标签（LLM评审员）：由 Claude 4.5 Haiku 生成，提供恢复、传播、不确定性等标签，作为辅助解释信号（已验证与人类标注的一致性）。

### 算法流程（文字说明）
Algorithm 1 从清洁运行开始，然后将故障注入运行，收集所有未通过的确定性检查；如果提供缓解措施，则用相同故障重新运行缓解后的智能体，并检查是否所有先前失败的检查都通过（`fix_confirmed`）。

## 6. 主要结论与发现

- **性能差距**：最佳智能体 DeepSeek 通过 105/120（87.5%），最弱 Llama 仅 77/120（64.2%）。这种差距在理想条件下的基准中不会出现。
- **弱点集中于数据质量故障**：所有智能体在 B 类（陈旧、矛盾、错误、空结果）上表现最差，即使最强 DeepSeek 也只得到 29/40。这些故障无错误码，容易遗漏。
- **缓解效果差异**：
  - **重试**显著改善工具执行故障（A1 超时从 3/10→10/10，A2 和 A3 也有提升）。
  - **数据质量故障**（B1-B3）几乎不受输出层缓解的影响，需更深入的机制（如时间戳检查、交叉引用）。
  - **安全故障**（C 类）在测试中基线得分就较高（10/10 或 7/10），缓解并未显示显著提升（C4 不变）。

## 7. 优点

- **填补空白**：同时提供了故障注入（如基准）和可干预、可验证的修复循环（如监测工具没有的），整合成一个统一工作台。
- **MCP 作为干预表面**：利用 MCP 协议通用性，允许开发者测试自己的智能体和服务器。
- **12 种现实故障类型**：分类清晰，涵盖工具执行、数据质量、安全三类，共 120 个场景，来源多样化（部分来自成熟基准，部分新创建）。
- **评分双轨制**：确定性检查为主判决，LLM评审员为辅助解释信号，且通过了人类验证（Cohen's κ 大部分>0.69）。
- **可视化交互**：提供 Web 界面（图 2），直接比较清洁/故障/缓解轨迹，定位分歧点。
- **缓解验证闭环**：能自动确认特定故障是否被缓解，打破了观察后手动再分析的低效模式。
- **开源**：代码、场景、评分器、结果均以 MIT 许可证发布。

## 8. 不足与局限

- **单故障注入**：仅一次一个故障，未评估复合级联或连续故障，这是与真实场景的主要差距。
- **故障类型不全面**：12 种故障未覆盖所有可能故障模式（如长时域依赖、政策违规、协调失败）。
- **B 类（数据质量）场景多为作者创作**，其难度和代表性未经过外部基准验证，结论应视为假设。
- **LLM 评审员不可靠**：作者明确警告 LLM-as-judge 可能会误标，传播率应视为上界。这降低了诊断标签的可信度。
- **确定性检查有模式限制**：可能无法覆盖所有合理修复行为或边缘情况。
- **缓解实验范围有限**：仅在一个弱模型（Llama）上测试，且未探索更深入的缓解（如时间推理、交叉引用），因此修复成功率为下限。
- **可能产生虚假信心**：`fix_confirmed` 标签仅表明一个特定故障不再触发检查，并非通用鲁棒性认证。
- **使用商业API**：依赖第三方模型提供者，可能引入不可控因素，且未报告精确算力消耗（如 token 数）。
