## 摘要
LLM智能体跨越命令行界面（如Codex）和智能体框架（如LangChain），将后端LLM与解析模型输出、控制智能体循环及管理上下文的框架代码相集成。框架与LLM生成的响应共同塑造智能体的执行过程。这种架构导致了无法通过单独检查任一组件而轻易理解的缺陷，因为某些缺陷仅在特定LLM响应引发智能体异常反应时出现。以往对智能体缺陷的实证研究主要将失败归因于模型能力有限或框架侧缺陷（如过时API和配置不匹配），而未对这些反应性缺陷进行描述。我们进行了首个聚焦于智能体反应性（AR）缺陷的实证研究。通过对来自Codex、Gemini-CLI、LangChain和CrewAI的255份缺陷报告进行人工分析，我们构建了一个涵盖可观察症状及其触发LLM行为的双轴分类体系。我们的发现表明，许多AR缺陷表现为没有明确定义测试预言的静默错误，这使得检测变得困难。LLM响应的随机性进一步复杂化了缺陷复现。我们还检查了用户提出的和开发者实现的修复方案。这一分析揭示了不匹配：用户经常倡导在框架侧设置护栏，而开发者可能将问题归因于LLM或对用户提出的修复方案响应缓慢。这些发现指向了对帮助用户和开发者理解AR缺陷根本原因和解决方案的机制的需求。总体而言，该研究强调了LLM智能体特有的挑战，并为AR缺陷的测试预言设计、复现支持及故障定位技术提供了动机。

## Abstract
LLM agents span command-line interfaces (e.g., Codex) and agent frameworks (e.g., LangChain), integrating backend LLMs with harness code that parses model outputs, controls agent loops, and manages context. Both the harness and LLM-generated responses jointly shape an agent's execution. This architecture gives rise to bugs that cannot be readily understood by inspecting either component alone, because some bugs occur only when a particular LLM response elicits an abnormal reaction from the agent. Prior empirical studies of agent bugs have largely attributed failures either to limited model capabilities or to harness-side defects, such as outdated APIs and configuration misalignment, without characterizing these AR bugs. We conduct the first empirical study focused on agent-reactive (AR) bugs. Through manual analysis of 255 bug reports from Codex, Gemini-CLI, LangChain, and CrewAI, we construct a two-axis taxonomy covering observable symptoms and the LLM behaviors that trigger them. Our findings show that many AR bugs manifest as silent errors without well-defined test oracles, which makes detection difficult. The stochasticity of LLM responses further complicates bug reproduction. We additionally examine fixes proposed by users and implemented by developers. This analysis exposes a mismatch: users frequently advocate harness-side guardrails, whereas developers may attribute the issue to the LLM or respond slowly to user-proposed fixes. These findings point to the need for mechanisms that help users and developers understand the root causes and resolutions of AR bugs. Overall, the study highlights challenges specific to LLM agents and motivates the design of test oracles, reproduction support, and fault-localization techniques for AR bugs.

## 1. 论文的核心问题与整体含义（研究动机和背景）

LLM智能体（如Codex、Gemini-CLI等命令行界面，以及LangChain、CrewAI等框架）由**后端LLM**和**框架代码（Harness）** 两部分组成。框架负责解析模型输出、控制智能体循环、管理上下文等，而LLM生成响应。两者交互共同决定智能体的执行路径。

现有研究将智能体故障主要归因于：① LLM能力不足（如幻觉）；② 框架侧缺陷（如过时API、配置不匹配）。但这两类视角忽略了**模型与框架交互**所引发的独特缺陷——即**代理反应性（Agent-Reactive, AR）缺陷**：只有特定LLM响应与框架处理方式共同作用时才显现的故障。例如，子智能体忽略框架附加的交接指令，反而递归生成子智能体。

**研究动机**：AR缺陷无法通过单独检查LLM或框架来诊断，且LLM的随机输出使复现困难。因此，有必要系统理解AR缺陷的症状、触发机制及修复策略，为LLM智能体的可靠性提升提供指导。

## 6. 论文的主要结论与发现

### 症状（RQ1）
| 症状 | 数量 | 特点 |
|------|------|------|
| 静默错误 (SE) | 108 | 占比最大；智能体输出流畅但结果错误，无异常或警告，用户需检查工具日志、工作区变化才能发现 |
| 崩溃 (Crash) | 71 | 抛出异常，会话终止或无法继续 |
| 输出错误 (EIO) | 34 | 会话仍在运行，但显示原始错误消息（如`NoneType AttributeError`） |
| 重试循环 (RL) | 33 | 智能体重复相同动作直至达到限制或用户干预 |
| 挂起 (Hang) | 9 | 智能体无响应或不产生可见进展 |

### 触发行为（RQ2）
| 行为 | 数量 | 主要对应症状 |
|------|------|-------------|
| 任务指令不遵从 (INC) | 67 | 81%转化为静默错误 |
| 意外工具参数 (UTA) | 62 | 取决于框架捕获层：LangChain多崩溃，Gemini-CLI多重试循环 |
| 消息模板冲突 (MTC) | 49 | 框架解析器失败时多崩溃 |
| 不成熟的框架接口 (IHI) | 23 | 如进程管理缺陷导致挂起 |
| 声明编造 (SF) | 20 | 90%为静默错误 |
| 空响应 (EmpR) | 12 | 症状取决于框架如何处理空载荷 |
| 上下文溢出 (CO) | 11 | 可转为静默错误或崩溃 |
| 工具幻觉 (TH) | 11 | 使用未定义工具名，多崩溃或EIO |

### 修复目标态度（RQ3）
- **用户**倾向于框架侧修复：45%的问题附带用户创建的PR，19%直接抱怨框架。
- **开发者**：24%合并了框架侧PR，15%明确归因于LLM，46%参与但未明确修复目标，15%没有参与。
- **不匹配**：约7%的问题出现用户建议框架修复、开发者却归因于LLM的情况，集中在INC和SF两类行为。例如LLM忽略指令或编造声明时，用户希望框架增加护栏，开发者认为“只能改进模型”。

### 核心挑战
- AR缺陷难以检测：大多表现为静默错误，缺乏严格预言。
- 复现困难：依赖特定的LLM输出、上下文长度、工作区状态和模型版本。
- 故障定位争议：用户和开发者对根本原因看法不一致。

## 7. 优点：方法或实验设计上的亮点

1. **首次系统研究AR缺陷**：填补了以往研究忽略“模型-框架交互”这一空白。
2. **严格的数据过滤与人工标注**：通过多层次自动化+人工编码，确保标注质量；两位作者独立标注并达成共识，减少主观偏差。
3. **双轴分类体系**：症状×触发行为，既能描述用户感知，又能揭示框架处理方式的差异如何导致不同症状，为故障定位提供结构化视角。
4. **修复目标的双向分析**：同时考察用户提议和开发者行动，揭露了二者在故障归因上的错位，具有实践指导意义。
5. **跨项目比较**：覆盖CLI产品与框架，发现不同项目在某些症状（如崩溃 vs 静默错误）上的分布差异，有助于理解架构设计对缺陷表现的影响。

## 8. 不足与局限

1. **样本偏差**：仅选取4个开源项目，且只分析高讨论度问题（3,037/11,288），可能忽略大量低参与度的AR缺陷。闭源项目（如Claude Code）无法纳入。
2. **人工标注的主观性**：症状、触发行为的分类依赖于标注者对issue文本的理解，可能存在歧义。尽管采用双人标注+讨论，仍无法完全避免。
3. **外部有效性有限**：4个项目可能不代表所有LLM智能体类型（如多智能体协作场景、专用领域智能体）。
4. **未评估修复效果**：论文仅记录修复类型，未验证合并的PR是否真正解决了缺陷，也未量化修复后的可靠性提升。
5. **缺乏自动化工具支撑**：分类和标注完全手工，未提出自动化分类器或检测工具，实用性受限。
6. **缺失算力和成本分析**：未讨论LLM调用次数、token消耗、计算成本等，可能影响修复方案的经济可行性评估。
