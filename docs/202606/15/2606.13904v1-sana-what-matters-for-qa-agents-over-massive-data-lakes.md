## 摘要
基于数据湖的探索性问答（EQA）要求一个LLM代理发现相关来源、分析检索到的数据，并根据中间结果调整其行为。仅凭端到端准确率无法区分搜索、规划、数据分析或代理的行动策略（即关于下一步做什么以及何时提交答案的决策）中的失败。我们提出了SANA（搜索代理导航消融框架），这是一个诊断性消融框架，它将EQA任务转化为包含黄金来源序列、清洗后的子问题以及执行记录的运行时轮廓。SANA利用这些轮廓构建理想化的搜索、规划和数据分析工具，从而允许对每个组件进行消融；剩余差距是诊断策略失败的证据。为了说明SANA作为一个可重用的评估框架，我们调整了两个最新的EQA基准：LakeQA和KramaBench，并在固定提示、预算、数据湖和运行时间下评估了轻量级和中型代理。在这两个基准中，数据分析始终是一个瓶颈，而规划则不那么突出。搜索在LakeQA的大数据湖设置中是一个主要限制，但在较小规模的KramaBench中不太显著。因此，SANA将端到端任务准确率解构为数据湖代理失败所在的诊断，并允许对搜索、规划、数据分析和代理设计方面的进展进行系统比较。

## Abstract
Exploratory question answering (EQA) over data lakes requires an LLM agent to discover relevant sources, analyze retrieved data, and adapt its actions based on intermediate results. End-to-end accuracy alone cannot distinguish failures in search, planning, data analysis, or the agent's Action Policy: its decisions about what to do next and when to submit an answer. We present SANA (Search Agent Navigation Ablation framework), a diagnostic ablation framework that transforms EQA tasks into runtime profiles containing gold source sequence, sanitized subquestions, and execution records. SANA uses these profiles to construct idealized search, planning, and data-analysis tools, allowing each component to be ablated; the residual gap is diagnostic evidence for policy failures.   To illustrate SANA as a reusable evaluation framework, we adapted two recent EQA benchmarks, LakeQA and KramaBench, and evaluated lightweight and mid-sized agents under fixed prompts, budgets, data lakes, and runtimes. Across both benchmarks, data analysis is a consistent bottleneck while planning is less so. Search is a major limitation in LakeQA's large data-lake setting, but less so for the smaller-scale KramaBench. SANA thus deconstructs end-to-end task accuracies into a diagnosis of where data-lake agents fail, and allows for systematic comparisons of progress in search, planning, data analysis, and agent design.

### 1. 论文的核心问题与整体含义
论文关注的是在大型数据湖上进行**探索性问答（EQA）** 时，LLM Agent 需要在多步交互中自主发现相关数据源、分析数据并调整后续行动。现有的端到端准确率指标只能反映整体成败，却**无法定位失败的具体环节**——是搜索不到关键数据、规划分解错误、数据分析代码有 bug，还是 Agent 自身的“行动策略”（何时搜索、何时提交答案）出了问题。作者提出 SANA 框架，通过组件级消融实验，为 EQA 代理提供细粒度的诊断分析。

### 2. 方法论
- **核心思想**：将 EQA 任务转化为一个“运行时轮廓”（runtime profile），包含黄金数据源序列、清洗后的子问题以及经过验证的执行记录。基于该轮廓，SANA 为搜索、规划、数据分析三个组件分别构造**理想化版本**（oracle 级工具），从而隔离这些组件的贡献。
- **关键技术细节**：
  - **任务轮廓**：对每个 EQA 任务，标注出 (1) 黄金数据源顺序；(2) 去除了数据集名称信息的子问题序列（避免泄露）；(3) 每个需要分析的子问题对应的正确数据集、分析意图、正确 SQL/Python 代码以及中间答案。
  - **理想化工具**：
    - **理想搜索**：只从黄金数据源集合 `D_gold` 中检索，返回与查询意图匹配的数据集（由一个 selector sub-agent 判断）。
    - **理想数据分析**：若 Agent 的意图与执行记录中的分析意图匹配，直接返回正确中间答案；否则使用更强大的模型（gpt-5.4）修复代码并执行，最多重试两次。
    - **理想规划**：将清洗后的子问题序列预先提供给 Agent，并给出更清晰的提示，引导 Agent 显式写出步骤和目标。
  - **消融方式**：每次只替换一个组件为理想版本，其他组件保持默认或标准实现，观察准确率变化。剩余的性能差距归因于 Agent 的行动策略（policy）失败。
- **算法流程**：Agent 在 reason-act-observe 循环中使用工具；SANA 作为框架注入理想化或标准工具，记录轨迹和最终答案，然后进行语义匹配评估和失败分析。

### 3. 实验设计
- **数据集/基准**：
  - **LakeQA**（135 个任务）：数据湖规模约 4000 万文档，平均每个任务需约 6.9 个黄金数据源。
  - **KramaBench**（转换后 83 个任务）：原始数据集较小（1764 个源），平均每个任务需约 2.3 个黄金数据源。将原始脚本分解为 LakeQA 风格的多跳任务。
- **对比方法**：每个组件有多种级别：规划（无计划、标准计划、理想计划）；搜索（BM25、Pneuma 混合检索、理想搜索、预加载黄金源）；数据分析（标准代码生成、理想分析）。共形成 3（规划）×4（搜索）×2（分析）= 24 种组合（实际表格仅列出关键组合）。
- **Agent 模型**：弱模型 `gpt-5.4-nano`，强模型 `gpt-5-mini`。
- **固定条件**：最大工具调用预算 30 次，每次工具超时 150 秒，总运行 600 秒上限，使用相同的系统提示和 Strands Agent 框架，并加入对话压缩和策略提醒插件。

### 6. 主要结论与发现
1. **数据分析始终是瓶颈**：在两个基准上，理想化数据分析带来的增益最大（LakeQA 上 +8.2%~+18.5%，KramaBench 上 +14.5%~+24.1%）。强模型更容易到达正确意图但实现失败，弱模型则更多在意图选择上出错。
2. **搜索在大数据湖中极为关键**：在 LakeQA 上，理想搜索比 BM25 提升 13%~14%，预加载黄金源进一步提升（尤其对弱模型 +28.9%），说明弱模型在搜索导航（提出正确查询、承诺来源）上困难更大。在 KramaBench 上搜索重要性降低。
3. **规划影响较小**：弱模型生成的计划与理想计划相似度达 81.5%，但遵循计划的比例很低（仅 19%~28%）；强模型计划遵循性更好（~50%）。因此主要问题不在于分解能力，而在于执行时偏离计划。
4. **残余策略失败**：即使所有组件理想化，仍留下约 24%~63% 的错误率（因模型和任务而异），这些归因于 Agent 的行动策略问题，如过早提交、工具重复循环、最终答案合成错误等。
5. **不同基准瓶颈不同**：LakeQA 侧重搜索，KramaBench 侧重数据分析，说明框架可用于刻画不同任务情境下的主要矛盾。

### 7. 优点
- **诊断性框架**：首次系统化地将 EQA 失败拆解为搜索、规划、分析、策略四个层次，并提供了可复用的消融工具。
- **方法论创新**：通过意图保持和理想化工具巧妙地隔离了组件效果，避免了因更换工具而改变 Agent 行为的偏差。
- **基准适应性**：成功将 KramaBench 转换为 LakeQA 风格任务，验证了框架的通用性。
- **结果可解释性**：不仅报告准确率，还通过失败审计揭示了不同模型的具体失败模式（弱模型广泛失败 vs 强模型集中于计算和最终答案）。

### 8. 不足与局限
1. **理想化工具的近似性**：理想搜索仅从标注的 `D_gold` 中选取，可能遗漏其他合法有效的数据源；理想数据分析依赖语义匹配，存在误配或漏配的可能。因此诊断上界是近似的。
2. **任务套件有限**：只评估了 LakeQA 和转换后的 KramaBench，无法直接推广到所有 EQA 场景（如多模态数据湖、流式数据等）。
3. **LLM 判断的偏差**：语义匹配、计划轨迹审核、失败标签均由 LLM 完成，可能引入 false positive/negative 和 label 偏差。
4. **行动策略的量化不足**：虽然指出了残余策略失败，但没有进一步细分策略失败的具体原因（如过度自信、缺乏回溯机制等），也未提出改进策略的明确方向。
5. **算力和成本分析缺失**：未报告每个任务的推理成本、token 消耗或运行时间，使得实际部署的可负担性不明确。
