## 摘要
AI智能体的性能不仅仅是一个建模问题，本质上是一个系统问题。模型的高级能力通过智能体框架（harness）得以实现。因此，模型假设与框架行为之间的差距很容易阻碍模型的全部能力转化为智能体性能。我们将此形式化为“意图-执行”差距：模型意图与框架执行之间的不匹配，反之亦然。我们认为，最小化这种意图-执行差距与框架设计的其他方面（如工具和执行循环）同样重要。为了说明这种框架-模型对齐的影响，我们开发了一个简单且可定制的框架，称为“简单链智能体”（SSA）。SSA旨在发现跨不同模型系列（如Claude、Gemini、GPT、Grok、Qwen）泛化的常见模式主体，以及少量特定于模型的偏好。我们做出两项贡献：（i）我们复现或改进了不同模型提供商系列在流行的智能体基准测试（SWE-Pro、SWE-Verified和Terminal-Bench-2）上报告的pass@1性能；（ii）基于对SSA生成的138k条轨迹的分析，我们超越了在顶尖模型之间往往相对平均的pass@1数值。通过用代码状态空间表示智能体轨迹，我们观察到了问题解决行为中的模型级差异。更细粒度的度量，如编辑频率、测试活动和阶段转换，揭示了各个模型在自主问题解决的不同阶段如何分配精力。

## Abstract
AI agent performance is not just a modeling problem, it is fundamentally a systems problem. The advanced capabilities of models are realized through agent harnesses. Therefore, a gap between model assumptions and harness behavior can easily prevent the model's full capabilities from translating into agent performance. We formalize this as the `intent-execution' gap: the mismatch between what the model intends and what the harness executes, and vice versa. We argue that minimizing this intent-execution gap is as important as other aspects of harness design such as tools and execution loops. To illustrate the impact of this harness-model alignment, we develop a simple and customizable harness called `Simple Strands Agent' (SSA). SSA aims to find the bulk of common patterns which generalize across different model families (such as Claude, Gemini, GPT, Grok, Qwen), as well as a small number of model-specific preferences. We make two contributions: (i) we reproduce or improve on the pass@1 performance reported by diverse model-provider families on popular agentic benchmarks (SWE-Pro, SWE-Verified and Terminal-Bench-2), and (ii) building on an analysis of 138k trajectories generated by SSA, we look beyond the pass@1 numbers which tend to be relatively even across frontier models. By representing agent trajectories in code state-spaces, we observe model-level differences in problem-solving behavior. Finer-grained metrics such as edit frequency, testing activity, and phase-transitions reveal how individual models allocate effort across different stages of autonomous problem solving.

#### 1. 论文的核心问题与整体含义（研究动机和背景）

*   **核心问题**：论文指出，当前AI编码智能体（Coding Agent）的性能瓶颈，不仅仅在于大语言模型（LLM）本身的原始能力，更在于模型与智能体框架（Harness）之间的不匹配。这种不匹配被形式化为 **“意图-执行（Intent-Execution）差距”**。
*   **“意图-执行”差距的定义**：这是一个双向问题。
    *   **正向差距**：模型意图做什么（如“修改文件A”）与框架实际执行了什么（如因解析错误，框架执行了“读取文件B”）之间的差距。
    *   **反向差距**：环境中实际发生了什么（如“测试失败”）与模型接收到的信息（如“测试输出被截断，未显示失败”）之间的差距。
*   **研究动机**：随着LLM能力的提升，框架的对齐变得至关重要。一个设计不当的框架会严重限制模型能力的发挥，导致即便模型本身更强，最终的智能体性能也可能与较弱模型的性能相近。当前许多工作通过定制化提示词、专有工具或手工设计的执行图来减少这种差距，但这些方法对特定模型和基准的“过度拟合”导致其在新模型上失效，性能提升是脆弱的。
*   **整体含义**：论文主张，智能体性能本质上是一个**系统问题**。提升性能不能只关注模型，更应致力于设计一个好的、能够最小化“意图-执行”差距的通用框架。这需要找到跨模型家族都有效的**不变接口原则**，并仅对模型确实不同的地方进行少量适配。

#### 2. 论文提出的方法论：核心思想、关键技术细节

*   **核心思想**：提出一个名为**Simple Strands Agent (SSA)** 的通用、可定制化智能体框架，其设计核心目标是**最小化“意图-执行”差距**。SSA通过提供反馈丰富的工具边界、平衡推理与工具调用、以及对齐模型偏好的方式来实现这一目标。
*   **关键技术细节**：
    *   **SSA架构**：一个简单的ReAct循环框架，包含任务、提示词、模型接口、工具、执行环境（Docker）和钩子（Hooks）。所有模型的执行核心循环是相同的，仅通过模型适配器和少量提示词/工具差异进行适配。
    *   **反馈丰富的工具边界**：
        *   **结构化执行反馈**：所有Shell命令都会返回状态码、输出和错误信息，长输出保留首尾以防关键信息丢失。
        *   **更安全的编辑语义**：所有编辑操作都锚定到相邻行，拒绝部分行或重复匹配，并返回统一差异（diff），减少模型验证需求。
    *   **平衡推理与工具调用**：
        *   引入“推理助推”（Reasoning Nudge），鼓励模型“思考足够，然后验证”，优先使用工具获取环境事实，而非进行冗长的链式思考。
        *   对Claude系列，使用**定量目标**（如“进行50次以上工具调用”）来打破其过长的推理链条；对Gemini和Grok系列，使用**灵活指令**（如“尽可能多地使用工具”），因为定量指令会导致它们为“达标”而进行空调用。
    *   **对齐模型偏好**：
        *   承认不同模型在调用工具时有不同偏好（如GPT偏好`apply_patch`格式，Grok偏好原子化操作）。SSA通过模型适配器来满足这些偏好，而不是强迫所有模型使用统一接口。
    *   **轨迹质量分析：解决距离（Solution Distance）**：
        *   为了解决`pass@1`无法区分良好和低效求解轨迹的问题，提出了一个**文本级代码状态空间**和**解决距离**度量。
        *   该方法将智能体的每次编辑视为在代码状态空间中的一次移动，通过与一组已知正确修复方案的文本差异特征集（`phi(x)`）进行比较，计算当前仓库状态到最近正确解的距离`D(t)`。
        *   `D(t)`从1开始，降至0表示完全复现了一个正确解。`∆D>0` 则成为衡量**回溯（Backtracking）** 行为的量化指标，如`回溯编辑比率`（具有`∆D>0`的编辑在所有编辑中的占比）。

*   **评估协议**：
    *   每个模型在每个基准上运行**5次**，报告`pass@1`的平均值和95%置信区间（基于所有试次计算）。
    *   严格控制评估环境（禁用SWE基准的互联网访问，一致的Docker镜像）。特别地，论文还识别并修复了SWE-Bench-Pro公共容器的一个**Git历史泄露**问题，并进行了“净化（Sanitized）”评估以保证结果公平。

#### 4. 资源与算力

*   论文中并未明确提及训练该SSA框架或评测所用的GPU型号、数量和具体时长。
*   硬件和部署信息：
    *   所有运行使用**AWS PCS集群**的 **c7.48xlarge 实例**（计算密集型实例，非GPU专用）。
    *   最大并发数为**10**。
    *   Claude模型通过**Amazon Bedrock**服务；OpenAI、Gemini、Grok通过其商业API。
    *   开源模型（如Qwen系列、GPT-OSS）通过**vLLM**在内部SLURM集群上的 **p4.24xlarge** 实例进行自托管推理。论文附录F给出了每个模型的vLLM服务命令，例如`tensor-parallel-size 8`（使用8张GPU进行张量并行）。
    *   **总结**：论文未提供总训练或评估的GPU小时数，但指出使用了高性能计算资源和云推理服务，涉及的计算量是显著的。

#### 5. 实验数量与充分性

*   **实验数量**：
    *   **总量**：在3个基准上评估了**21个模型**，每个模型运行5次，共产生了约 **138,000 条高质量轨迹**（`(500 + 731 + 89) x 5 x 21`）。
    *   **评估维度**：不仅报告`pass@1`，还基于轨迹分析报告了**编辑/测试比率**、**阶段分布**、**回溯编辑比率**、**工具错误率**等多种细粒度指标。
*   **充分性与公平性**：
    *   **充分性**：实验覆盖了5个领先模型家族的几乎所有最新版本，轨迹数量巨大，分析维度丰富，具有很强的说服力。
    *   **公平性**：
        *   论文严格遵循评估规则（如禁止互联网访问）。
        *   特意指出并量化了**SWE-Bench-Pro的Git历史泄露问题**，对比了“Base”和“Sanitized”两种环境下的分数，证明了该问题的存在和高影响力，体现了评估的严谨性。
        *   在所有基准上使用**统一的SSA框架**，仅有极少的模型家族适配，避免了为特定模型定制框架而带来的不公平优势。
        *   报告了**95%置信区间**，增加了结果的可信度。

#### 6. 论文的主要结论与发现

1.  **量化“意图-执行”差距的影响**：通过SSA框架，在多个基准上复现或超过了绝大多数模型的官方报告`pass@1`结果。这证明`pass@1`的差距很大程度上源于框架摩擦，而非模型能力的根本限制。
2.  **轨迹行为是比`pass@1`更富信息的模型签名**：不同模型家族即使在`pass@1`非常接近时，其解决问题的轨迹（如编辑频率、测试活动比例、阶段转换模式）也**截然不同**，这些是稳定的、特定于模型的行为特征。
    *   例如，Opus 4.6在解决SWE-Bench任务时，更早、更多地投入测试，而GPT-5.4的编辑和测试活动在时间分布上不同。
    *   **回溯行为**与**解决率**脱钩。Gemini 3.1 Pro拥有最高的回溯率（14.3%），但同时拥有最高的解决率（80.8%）。而GPT-5.4回溯率极低（1.2%），解决率也在顶尖之列（80.2%）。这表明“如何解决问题”与“能否解决问题”是两个独立的维度。
3.  **最小化意图-执行差距是关键**：不需要复杂的任务特定调优，只需做好反馈、对齐和平衡，一个简单的ReAct循环（SSA）就能在多种模型上达到顶尖效果。
4.  **评估基础设施影响重大**：Terminal-Bench-2的结果受计算和内存限制影响显著；SWE-Bench-Pro的**Git历史泄露**能显著`inflate pass@1`（最多可达6.87%），且通过明确的git指令提示，泄露效应可以放大17%以上。

#### 7. 优点（亮点）

*   **视角创新**：将智能体性能问题从“模型端”转移到“模型-框架接口端”，提出了“意图-执行”差距这一新颖且有洞见的理论框架，具有很高的指导价值。
*   **分析深度**：超越了单一的`pass@1`指标，通过构建“代码状态空间”并提出“解决距离”、“回溯”等量化指标，对智能体的**轨迹质量**进行细致分析，揭示了被宏观指标掩盖的模型行为差异。
*   **深入骨髓的公平性**：不仅自身实验设计严谨，还主动揭示并量化了公共基准的潜在漏洞（Git历史泄露），这是很负责任的学术行为。
*   **可复现与实用性**：开源的SSA框架设计简洁，易于定制和扩展。论文详细记录了所有配置、提示词和工具定义，为社区提供了极有价值的基线框架。
*   **大规模跨模型分析**：对21个模型在3个基准上进行了统一分析，结论具有很强的泛化能力。

#### 8. 不足与局限

*   **分析基于文本，而非语义**：“解决距离”的度量基于文本差异。这意味着一个“正确但文本表达不同”的修复方案（如变量名不同、逻辑重组）可能会被误判为“距离较远”，其精度受限于观察到的经验性参考集。
*   **回溯的语义解读有限**：目前的回溯度量（`∆D>0`）将所有远离参考解的行为都视为负向回溯。但如文中所述，一些回溯可能是故意的A/B测试，甚至能发现第二个Bug，这些**有益的回溯**与无效的“找到又丢失”行为被归为一类，无法区分。
*   **状态重建的保真度**：基于工具调用解析重建的状态`d(t)`无法完美捕捉通过复杂Shell命令（如Python脚本修改文件）造成的所有状态变化，其精度存在上限。
*   **模型家族适配的边界**：虽然声称“单一框架”，但文中揭示，在提示词（如推理助推策略）和工具接口（如`apply_patch` vs `str_replace_editor`）上，不同模型家族**确实需要不同的适配**。这表明“通用的最佳实践”目前尚不存在，仍需经验性地探索每个模型家族的偏好。
*   **未覆盖所有模型类型**：实验虽然覆盖了主流家族，但未包含如开源模型DeepSeek、Llama等，结论的泛化到所有模型上仍有待验证。
