## 摘要
我们提出了GRACE-DS，一个用于数据科学的受保护奖励引导的智能体修正环境，用于对基于LLM的AutoML智能体进行部署前评估。GRACE-DS是一组在隔离环境中的评估指标，可应用于特定组织特有的表格机器学习任务。它将智能体暴露于真实的工作流阶段，从规划和数据检查，到特征工程、模型开发、验证、代码修复，直至最终提交，而隐藏的可执行验证器不仅衡量最终的预测性能，还衡量泄露避免、可复现性、协议有效性、修正行为和奖励对齐。最强结构化的机制——灵活的迭代交互（我们的方法）——在端到端归一化隐藏测试质量上优于单次生成、非结构化交互和基于重启的基线，同时提高了协议有效完成率。在超过7000个回合中验证的结果表明，GRACE-DS是一个稳健的平台，用于评估基于LLM的AutoML智能体在类似生产条件下并按照组织特定要求执行机器学习工作流的能力。

## Abstract
We introduce GRACE-DS, a Guarded Reward-guided Agent Correction Environment in Data Science for pre-deployment evaluation of LLM-powered AutoML agents. GRACE-DS is a set of evaluation metrics in an isolated environment that can be applied to tabular ML tasks specific to a particular organization. It exposes agents to realistic workflow stages, from planning and data inspection through feature engineering, model development, validation, and code repair to final submission, while hidden executable validators measure not only final predictive performance but also leakage avoidance, reproducibility, protocol validity, correction behavior, and reward alignment. The strongest structured regime, flexible iterative interaction (our approach), achieves higher end-to-end normalized hidden-test quality than single-shot generation, unstructured interaction, and restart-based baselines, while also improving protocol-valid completion. Validated across more than 7,000 episodes, these results establish GRACE-DS as a robust platform for assessing the capacity of LLM-based AutoML agents to execute machine learning workflows under production-like conditions and in accordance with organization-specific requirements.

# GRACE-DS 论文详细总结

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有的 AutoML 智能体（LLM-powered）评估主要依赖最终得分（如 Kaggle 排行榜），但企业部署时需要更全面的诊断：不仅要“是否取得了好结果”，更要“如何取得该结果”，是否伴随数据泄漏、无效验证、不可复现等不可信行为。因此需要一种**预部署评估平台**，能够模拟生产环境下的完整工作流，同时测量过程可靠性（如泄漏规避、可复现性、协议有效性、修正行为）和最终预测性能。
- **背景**：表格机器学习仍广泛用于工业应用（风控、需求预测、欺诈检测等），数据敏感且组织特定。现有基准（MLE-bench、MLGym、DSGym 等）要么只报告最终分数，要么不聚焦表格 ML 和部署约束。GRACE-DS 填补了这一空白，提供受控的隔离评估环境，用于评估 LLM 驱动的 AutoML 智能体在类似生产条件下的能力。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：构建一个**可控的、阶段感知的交互环境**，将表格 ML 任务分解为八个状态（PLAN、EDA、FEATURE ENGINEERING、MODEL、VALIDATE、CODE、CODE FIX、FINAL SUBMIT），每个状态有独立的步骤、时间和重复预算。智能体通过 action: 指令交互，执行 Python 代码片段。评估分为两部分：隐藏的可执行验证器（5 组，包括完整性/泄漏、数据质量、建模、过程、终端比较）和一次性的隐藏测试评分。
- **关键技术细节**：
  - **环境作为部分可观察马尔可夫决策过程 (POMDP)**：状态包含当前阶段、工作空间、真实任务质量、隐藏评估规则；智能体仅能感知结构化反馈。
  - **奖励函数**：分解为性能项 \(r_{perf}\)（权重 0.55）、计划覆盖项 \(r_{plan}\)（0.15）和代码质量项 \(r_{code}\)（0.30）。软惩罚上界为加权奖励的 60%，关键错误（如目标泄漏）强制奖励为零。
  - **验证器套件**：包括完整性验证、泄漏检测（代码模式 + 行为分析）、数据条件检查（缺失值、类别等，基于数据真实情况动态激活）、可复现性检查、模型与效率约束（禁止自动超参搜索）、过程验证（计划覆盖、回溯、迭代改进）。输出仅给出抽象提示（不暴露验证规则细节）。
  - **隐藏测试评分**：最终提交后，由评估方独立复现管道并在隐藏测试集上评分一次，该分数与过程奖励分离，防止奖励黑客。
  - **灵活迭代交互（flexible-iterative）**：智能体自主选择阶段顺序，但需遵守“候选优先”规则（先构建一个简单基线的可复现模型，再进行深度 EDA；同时鼓励至少两个不同族的模型候选），并通过轻量正则表达式多样性信号奖励。
  - **对比机制**：包括单次生成（single-shot）、无结构智能体（unstructured-agent，有工作空间和反馈但无阶段标签）、重启动基线（从零开始或调用匹配上限）、固定阶段迭代（fixed-stage-iterative，强制顺序）等。

## 3. 实验设计：数据集、基准、对比方法

- **数据集**：10 个生产风格任务，分四个家族：
  - TML-Bench：4 个后截止 Kaggle 竞赛
  - TabReD：1 个工业数据集
  - UCI/OpenML：3 个较少使用的基准
  - 合成数据：2 个已知数据生成过程的表格任务（零记忆风险）
- **基准/对比方法**：评估了 15 种机制，包括 7 个核心机制（列表见论文表 18），5 个留一状态消融，2 个奖励优化探测和 1 个红队对抗。核心基线包括：
  - single-shot：仅 MODEL 一次
  - unstructured-agent：有工作空间但无阶段标签和提示
  - restarts-from-scratch / restarts-call-matched：多次独立尝试
  - fixed-stage-iterative：强制顺序 PLAN→EDA→FE→MODEL→…→ refine
  - flexible-iterative：自主选择顺序，候选优先
- **指标**：
  - 端到端归一化隐藏测试质量（E2E Q）：未完成的视为 0。
  - 观测质量（Obs. Q）：仅针对完成提交的回合。
  - 协议有效完成率（PV）：可复现且一次终端评分、无协议违规。
  - 关键错误率、恢复率（Rec. any / Rec. exec.）、奖励对齐（Pearson/Spearman）、成本（秒、调用数、标记数）。

## 4. 资源与算力

- **模型训练和评分**：所有管道在 CPU 上运行（AMD Ryzen 7 5700X，64 GB RAM），8 个工作进程并行。

## 5. 实验数量与充分性

- **实验数量**：核心网格包括 8 个 LLM × 10 个任务 × 7 个核心机制 × 6 个重复（种子、温度、重复）= 480 回合/机制。加上 5 个消融（各 480 回合）、2 个奖励优化探测（各 480 回合）和 2 个模型对比的红队（120 回合），总计超过 7000 回合。详见表 4-12、19。
- **充分性与公平性**：
  - 所有机制共享相同的预算（步骤/时间/标记）、分种子、温度、重复，并进行了配对 Wilcoxon 符号秩检验和引导置信区间，统计充分。
  - 避免数据集记忆：使用后截止 Kaggle 和合成数据；隐藏测试评分严格由评估方控制；验证器规则不向智能体披露。
  - 消融实验区分了每个状态的具体贡献（如 PLAN 和 EDA 是支撑性的，FEATURE ENGINEERING 可以吸收进 MODEL）。

## 6. 论文的主要结论与发现

- **结构化迭代显著提升端到端质量和协议有效性**：flexible-iterative 的 E2E Q 达 0.754，比 single-shot（0.536）高 +0.218，比 unstructured-agent（0.527）高 +0.227，比 restarts-from-scratch（0.672）高 +0.082，比 call-matched restarts（0.686）高 +0.068，且 PV 高达 96.9%。
- **隐藏验证器有效防止关键方法论错误**：structured 机制中 critical-error 事件近 0（最多 1/480），而 restart 基线达 5-7%。红队对抗无法绕开验证器（0 次泄漏、0 次禁止访问、0 次快照篡改），但隐藏测试能揭示质量损失。
- **过程奖励是有用但不可替代的进展信号**：与 E2E Q 任务内 Spearman 均值 0.30，但跨任务不宜聚合；奖励优化探测仅小幅提升奖励、损失 3-5% E2E Q。
- **EDA 和 PLAN 是支撑性阶段，FEATURE ENGINEERING 可被吸收**：留一消融中移除 EDA 或 PLAN 显著降低质量和 PV，移除 FE 几乎无影响。
- **模型规模影响使用环境的方式**：所有模型均受益于结构化环境，但较弱模型在固定阶段迭代中 PV 崩溃（如 Qwen3-Next-80B 仅 38.3%），灵活迭代恢复；最强模型受益相对较小但进程更稳定。
- **Q-S-C 前沿明确**：flexible-iterative 在质量上优于所有其他迭代机制，同时更快（平均 101s vs 120s unstructured）且调用更少；紧凑反馈变体可节省令牌但牺牲少量质量。

## 7. 优点

- **全面的过程验证**：不仅测量最终结果，还追踪泄漏、可复现性、协议遵循、修正行为、多元性等，产生可审计的诊断。
- **阶段感知的交互设计**：显式建模从计划到提交的完整工作流，隐藏验证器根据不同阶段动态激活，更符合生产实践。
- **奖励分解与隐藏测试分离**：防止奖励黑客，同时提供有意义的进展信号。
- **公平且可控的实验设置**：预算、种子、重复等变量严格固定，配对统计检验保证结论可靠；消融实验隔离了每个组件的贡献。
- **实际部署导向**：强调可复现的原始输入管道、保护敏感数据、符合组织特定验收标准。可直接将内部 `task.json` 和数据集投入评估。
- **可解释性**：提供终端路径分解、奖励分解、验证增长轨迹、候选多元性记录等，可诊断具体失败原因。
- **覆盖多种模型和任务**：混合专有和开源模型、不同架构和规模，增强结论泛化性。

## 8. 不足与局限

- **任务范围受限**：仅针对表格监督学习任务，未涵盖深度学习训练循环、多模态、强化学习、大规模分布式训练、架构搜索或开放式科学发现。论文承认这是未来工作方向。
- **治理覆盖不完整**：不包括 PII 检测、角色访问控制、数据保留与审计日志、公平性偏差审计、GDPR 等监管合规检查。只是选定的部署相关约束子集。
- **红队评估不穷尽**：仅基于固定提示的红队，未覆盖验证器感知攻击、手动或混淆的泄漏（如通过序列化、特征名称编码）、数据复制、从反馈推断验证标签等。论文声明不是彻底对抗审计。
- **不涵盖完整生产就绪性**：通过 GRACE-DS 不意味模型可投入生产，仍需要监控、校准、偏移健壮性、延迟约束、可解释性、安全审查和审批流程等。
- **过程奖励可能饱和**：在某些任务上性能项接近饱和后，奖励方差主要由过程项主导，对齐性变差（如合成回归任务 Spearman=-0.12）。论文已指出需按任务评估，但仍需注意。
- **未分析可复现管道在实际部署中可能遭遇的依赖冲突**（沙箱已固定版本，但真实环境变化可能影响）。
- **LLM 推理成本差异大**：OpenRouter 价格和时延波动；论文未计算货币成本，仅给调用/标记数，实际部署需自行估测。
