## 摘要
AI智能体性能不仅仅是一个建模问题，从根本上说是一个系统问题。模型的先进能力是通过智能体框架（agent harnesses）实现的。因此，模型假设与框架行为之间的差距很容易阻碍模型全部能力转化为智能体性能。我们将此形式化为“意图-执行”差距：模型意图与框架执行之间的不匹配，反之亦然。我们认为，最小化这种意图-执行差距与框架设计的其他方面（如工具和执行循环）同样重要。为了说明这种框架-模型对齐的影响，我们开发了一个简单且可定制的框架，称为“Simple Strands Agent”（SSA）。SSA旨在找出跨不同模型家族（如Claude、Gemini、GPT、Grok、Qwen）泛化的常见模式的大多数，以及少量模型特定的偏好。我们有两个贡献：（i）我们复现或改进了不同模型提供商家族在流行智能体基准测试（SWE-Pro、SWE-Verified和Terminal-Bench-2）上报告的pass@1性能；（ii）基于对SSA生成的13.8万条轨迹的分析，我们超越了在前沿模型中相对均匀的pass@1数字。通过在代码状态空间中表示智能体轨迹，我们观察到问题解决行为中的模型级差异。更细粒度的指标，如编辑频率、测试活动和阶段转换，揭示了单个模型在自主问题解决的不同阶段如何分配努力。

## Abstract
AI agent performance is not just a modeling problem, it is fundamentally a systems problem. The advanced capabilities of models are realized through agent harnesses. Therefore, a gap between model assumptions and harness behavior can easily prevent the model's full capabilities from translating into agent performance. We formalize this as the `intent-execution' gap: the mismatch between what the model intends and what the harness executes, and vice versa. We argue that minimizing this intent-execution gap is as important as other aspects of harness design such as tools and execution loops. To illustrate the impact of this harness-model alignment, we develop a simple and customizable harness called `Simple Strands Agent' (SSA). SSA aims to find the bulk of common patterns which generalize across different model families (such as Claude, Gemini, GPT, Grok, Qwen), as well as a small number of model-specific preferences. We make two contributions: (i) we $\textbf{reproduce or improve on the pass@1}$ performance reported by diverse model-provider families on popular agentic benchmarks (SWE-Pro, SWE-Verified and Terminal-Bench-2), and (ii) building on an $\textbf{analysis of 138k trajectories generated by SSA}$, we look beyond the $\texttt{pass@1}$ numbers which tend to be relatively even across frontier models. By representing agent trajectories in code state-spaces, we observe model-level differences in problem-solving behavior. Finer-grained metrics such as edit frequency, testing activity, and phase-transitions reveal how individual models allocate effort across different stages of autonomous problem solving.

## 1. 论文的核心问题与整体含义

论文的核心问题是 **“意图-执行差距”**：在 AI 智能体系统中，大语言模型（LLM）的意图与执行框架（harness）实际执行的操作之间存在双向不匹配。一方面，模型期望执行某个操作，但框架可能因为解析错误、工具设计不当等而执行了不同的操作；另一方面，框架返回的执行结果也可能无法准确反映环境状态，导致模型基于虚假信息推理。这种差距使得模型的能力无法完整转化为智能体性能，即便基础模型在原始能力上更强，若框架未对齐，最终性能也可能与较弱模型无异。因此，最小化意图-执行差距与设计工具和循环同样重要。该工作强调智能体性能本质上是系统问题，而非单纯的建模问题。

## 2. 方法论

### 核心思想
开发一个简单、可定制的框架 **Simple Strands Agent (SSA)**，围绕意图-执行差距设计，通过反馈丰富的工具边界、平衡推理与工具调用、以及对齐模型偏好，使框架能够最小化该差距，并在不同模型家族间实现通用性和有效性。

### 关键技术细节

- **SSA 架构**: 包含任务、提示生成器、模型适配器、工具集、执行环境（Docker）和钩子（hooks）。核心是循环：调用模型 → 执行请求的工具 → 附加工具结果 → 重复。
- **反馈丰富的工具边界**:
  - `bash` 返回结构化的命令、状态、退出码、输出、错误文本；长输出截断时保留首尾。
  - 编辑工具（`str_replace_editor`）使用行锚定，拒绝模棱两可的匹配，成功返回统一 diff，减少模型验证负担。
- **平衡推理与工具调用（Reasoning Nudge）**:
  - 鼓励模型进行足够推理以决定下一步，但优先通过工具与环境交互验证假设，而非长时间内部思考。
  - 对于 Claude 系列使用定量指令（如“进行 100 次以上工具调用”）；Gemini、Grok 等使用柔性指令（如“尽可能使用工具”）。
- **对齐模型偏好**:
  - GPT 系列倾向于 `apply_patch` 方式编辑代码；Grok 系列偏好原子化工具（拆分为 `file_read`、`file_edit`、`file_write`）且禁用行号显示。
- **文本级代码状态空间与 Solution Distance 度量**:
  - 定义代码状态空间 \(X\)，实例子空间 \(X_i\)，解集 \(S_i\)。
  - 对每个状态 \(x(t)\)，计算其统一 diff 特征集与最近参考解的特征集之间的召回散度：
    \[
    D_i(t) = 1 - \max_{x^* \in S_i} \frac{|\phi_i(x_i(t)) \cap \phi_i(x^*)|}{|\phi_i(x^*)|}
    \]
  - 通过忠实回放工具调用重构状态，计算每次编辑的 \(\Delta D\)，正值为回溯信号。

总体实验设计较为充分，覆盖主流模型家族和多个基准，并通过多次运行和多种控制实验验证结论的稳健性。

## 6. 主要结论与发现

1. **SSA 在 pass@1 上复现或超越官方成绩**：在所有三个基准上，SSA 的 pass@1 值均进入或高于模型官方报告分数的置信区间，证明最小化意图-执行差距足以获得高水准性能。
2. **轨迹行为是具有模型家族特异性的稳定签名**：
   - 即使 pass@1 相近（如 Opus 4.6 和 GPT-5.4 在 SWE-Verified 均约 80%），两者的编辑/测试比例、阶段分布、回溯模式差异显著。
   - GPT 系列回溯极少（<3%），Gemini 系列回溯频繁（14-17%），但 solve rate 同样优秀，说明 pass@1 无法反映搜索效率与行为稳定性。
3. **Solution Distance 曲线有效区分问题解决风格**：Opus 4.6 在 resolved 轨迹中快速下降，Gemini 3.1 Pro 的下降曲线更平缓且伴随中期回溯。
4. **识别并量化评估泄漏**：SWE-Bench-Pro 公共容器保留未来 git 对象，模型可通过 `git show` 暴露 gold patch SHA 而获得不公平优势，去除后 pass@1 下降最高 6.87%。
5. **基础设施敏感度**：Terminal-Bench-2 的结果受推理后端吞吐、依赖下载、命令批处理等系统因素影响，命令批处理对 Claude 无益（因增加推理复杂度），但对 Gemini 和 Grok 有利。

## 7. 优点

- **提出“意图-执行差距”** 这一系统级视角，将框架对齐提升到与工具设计同等重要的地位，为智能体设计提供了新框架。
- **SSA 设计简洁且通用**：核心工具集较小（主要依赖 bash 和编辑器），通过反馈丰富的边界和对齐模型偏好实现跨模型泛化，无需任务特调。
- **Solution Distance 度量创新**：超越二元 pass/fail，提供对搜索效率和轨迹质量的细粒度定量分析，支持回溯检测、到达时间等新指标。
- **大规模轨迹分析**（138k 轨迹）揭示了模型特定的行为签名，为模型选择和改进提供洞察。
- **开源所有代码、配置和提示**，重复性强。
- **发现并量化评估泄漏**，对未来基准设计和评估规范有警示意义。

## 8. 不足与局限

- **模型适配仍需手动**：虽然核心框架通用，但针对 Claude 需要定量工具调用提示，GPT 需要 `apply_patch` 工具等，并非完全自动化的对齐。
- **Solution Distance 基于文本特征**：可能忽略语义等价的不同实现；当参考解集稀疏时（超难实例），度量可靠性下降。
- **实验覆盖有限**：仅在三个编码基准上验证，未涉及 Web 代理、多模态、对话等通用智能体任务。
- **资源成本未深入优化**：仅给出输出 token 与 pass@1 的 Pareto 图，未系统探讨如何优化推理成本或延迟。
- **回溯定义依赖近似解集**：当真实解集未充分采样时，回溯检测可能不准确。
- **训练数据泄漏未检测**：仅检查了容器级别的 git 泄漏，未讨论模型预训练中可能接触过基准问题的情况。
- **工具错误率分析未考虑后端差异**：不同模型通过不同 API（Bedrock、vLLM、商业 API）调用，其固有延迟和解析错误差异可能影响轨迹表现，论文未深入分离这些系统因素。
