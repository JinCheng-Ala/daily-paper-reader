## 摘要
我们提出了一种用于主动容错控制（FTC）的智能大语言模型（LLM）框架，该框架将故障检测输出转化为基于工厂特定知识的约束感知恢复动作。该方法结合了（i）一个多智能体工作流，将操作员职责分解为监控、规划、动作合成、仿真、验证和重新提示；（ii）一个数字过程工厂孪生（DPPT），提供工厂数据、模型以及用于执行前测试的仿真服务；以及（iii）一个基于CPSMod本体构建的图检索增强生成（Graph RAG）层，该本体将工厂知识（结构、功能、混合动态、控制上下文和故障语义）组织成一个图，为智能体支持关系感知的多跳检索。纠正动作生成为最小风险的状态机恢复路径以及相应的离散命令或连续设定值调整，然后在任何执行前根据联锁、包络和动态可行性进行确定性验证。如果在有限时间窗口内未找到可接受的计划，则控制权移交给安全后备。该框架在仿真中针对两个代表性基准进行了评估：离散批处理混合模块和闭环PID调节下的连续搅拌釜反应器（CSTR）。使用轻量级LLM（GPT-4o-mini和GPT-4.1-mini）的结果表明，语义驱动的智能体能够在与各自过程动态相兼容的延迟预算内得出有效的恢复决策，为从检测到经过验证的纠正动作提供了一条实用路径，适用于离散和连续FTC任务。

## Abstract
We propose an agentic Large Language Model (LLM) framework for active Fault-Tolerant Control (FTC) that transforms fault detection outputs into constraint-aware recovery actions grounded in plant-specific knowledge. The approach couples (i) a multi-agent workflow that decomposes operator duties into monitoring, planning, action synthesis, simulation, validation, and reprompting; (ii) a Digital Process Plant Twin (DPPT) that exposes plant data, models, and a simulation service for pre-execution testing; and (iii) a Graph Retrieval-Augmented Generation (Graph RAG) layer built on the CPSMod ontology, which organizes plant knowledge (structure, function, hybrid dynamics, control context, and fault semantics) into a graph that supports relation-aware, multi-hop retrieval for the agents. Corrective actions are generated as minimal-risk state-machine recovery paths and corresponding discrete commands or continuous setpoint adaptations, then validated deterministically against interlocks, envelopes, and dynamic feasibility before any actuation. If no acceptable plan is found within a bounded time window, control is handed to a safety fallback. The framework is evaluated in simulation on two representative benchmarks: a discrete batch Mixing Module and a Continuous Stirred-Tank Reactor (CSTR) under closed-loop PID regulation. Results with lightweight LLMs (GPT-4o-mini and GPT-4.1-mini) show that semantically grounded agents can derive valid recovery decisions within latency budgets compatible with the respective process dynamics, demonstrating a practical pathway from detection to validated corrective action across both discrete and continuous FTC tasks.

## 1. 论文的核心问题与整体含义

**研究动机**：过程工业中，操作员依赖专业知识处理故障，但操作员能力不均、报警泛滥等问题突出。现有容错控制方法（如强化学习）存在样本效率低、泛化性差、奖励函数设计困难等局限，难以直接部署。大语言模型（LLM）在处理自然语言和复杂推理方面展现出潜力，但直接用于容错控制面临**知识接地**（plant-specific knowledge grounding）、**动作安全性保证**（pre-execution validation）和**实时性**三大挑战。

**核心问题**：如何构建一个能够自动将故障检测结果转化为经验证的、安全且高效的纠正控制动作的LLM智能体框架，并在批处理和连续过程控制场景中验证其有效性。

## 2. 论文提出的方法论

**核心思想**：将操作员职责分解为多个专用LLM智能体，利用数字孪生和基于本体的图RAG提供结构化领域知识，通过“规划-动作合成-仿真-验证-重新提示”的闭环流程生成并验证纠正动作。

**关键技术细节**：

- **多智能体架构**（Agent Layer）：
  - **监控智能体**（Monitoring Agent）：确定性/数据驱动组件，处理传感器数据与报警，检测故障并触发恢复流程。
  - **规划智能体**（Planning Agent）：LLM驱动的，根据故障症状和图RAG检索的状态机子图，枚举满足守卫条件的状态路径，选择最小风险恢复路径。
  - **动作智能体**（Action Agent）：LLM驱动的，根据规划路径和图RAG检索的执行器-组件子图，生成具体离散命令（如阀门开/关）或连续设定值调整（如温度/液位设定点）。
  - **仿真智能体**（Simulation Agent）：调用数字孪生的仿真服务，模拟动作执行后的系统动态轨迹和最终状态。
  - **验证智能体**（Validation Agent）：确定性规则引擎，检查仿真结果是否违反联锁、运行包络、动态可行性等约束，输出“批准”或“修订”及结构化反馈。
  - **重新提示智能体**（Reprompting Agent）：当验证失败时，根据结构化反馈或LLM生成的修正文本，将反馈注入到前序智能体的下一次提示中，形成迭代优化循环（最多n次，否则触发安全后备）。

- **工具层**（Tool Layer）：
  - **数字过程工厂孪生（DPPT）**：提供过程数据、物理/数据驱动模型、仿真服务。包含过程数据（传感器、致动器、报警等）、静态工程数据（P&ID、操作限制等）、仿真模型（基于物理/降阶模型）以及条件监控模型。
  - **图RAG**：基于**CPSMod本体**（整合VDI 2206、VDI 3682、UML State Machine、OpenMath、DIN 17359等标准设计模式），将结构、功能、行为、故障语义等信息组织为知识图谱。智能体通过预定义SPARQL模板检索相关子图，作为提示的补充上下文。支持关系感知的多跳检索。

- **信息流**：
  1. 监控智能体检测故障→生成故障类型与症状。
  2. 规划智能体根据故障和图RAG检索状态机，生成可行恢复路径。
  3. 动作智能体根据路径检索执行器子图，生成具体动作序列。
  4. 仿真智能体调用DPPT仿真动作结果。
  5. 验证智能体确定性地检查结果，若通过则执行，否则反馈给重新提示智能体，重试规划或动作步骤。
  6. 若迭代次数超限，则触发安全后备系统。

**算法流程**（文字描述）：
- 初始化：系统处于正常操作状态。监控智能体持续采集过程值，与统计基线比较。
- 若检测到偏差→状态设为“FAULT”并触发恢复工作流。
- 规划智能体通过SPARQL查询获取与故障相关的状态机片段。
- LLM基于该子图推理出从当前故障状态到目标安全状态的最优路径。
- 动作智能体通过SPARQL查询获取路径对应的执行器信息，LLM生成具体控制命令或设定值。
- 仿真智能体执行数字孪生仿真，返回轨迹与最终状态。
- 验证智能体检查仿真结果是否符合所有约束（联锁、包络、动态可行性等）。
- 若通过→批准执行；若未通过→重新提示智能体生成修正反馈，返回前序步骤重新规划或调整动作。
- 若在预算时间内无有效动作→安全后备接管。

## 6. 论文的主要结论与发现

1. **知识接地有效**：基于CPSMod本体的图RAG能够为LLM智能体提供足够的结构化知识，使其正确做出容错控制决策。GPT-4.1-mini在混合模块全部60次中100%成功，计划与动作正确率均为100%。
2. **仿真验证保障安全**：仿真智能体与验证智能体能够有效检测不可行动作，并通过重新提示机制纠正部分错误。但恢复率因故障类型而异，某些故障（如传感器故障、冷却故障）下GPT-4o-mini的恢复率很低（33%、0%），表明仅靠重新提示无法弥补根本性推理缺陷。
3. **轻量LLM可行**：GPT-4.1-mini在两种场景下均表现优异（CSTR成功率93.3%），单次决策延迟约2-3秒，整体恢复时间约30-50秒，对于慢速批处理和监管控制场景可接受。但无法用于毫秒级实时系统。
4. **错误模式不对称**：GPT-4o-mini的失败主要源于**额外行动**（false positive，如在不该启动时启动主泵）或**错误路径选择**（如选择旁路路径而非正常路径），但从未遗漏必要动作（召回率100%）。这表明语义接地能确保核心动作，但弱模型会引入冲突的“常识”推断。
5. **动作与连续性**：在CSTR结垢故障中，框架成功通过降低进料流量设定点来折中产量而维持稳定运行，表现出策略权衡能力。

## 7. 优点

- **结构化多智能体设计**：将容错控制任务划分为专属智能体，各司其职，LLM仅用于需要推理和语义理解的部分，而确定性组件负责仿真与验证，保持了可审计性和安全性。
- **结合知识图谱的图RAG**：直接利用图结构进行关系感知的多跳检索，优于普通文档级RAG，特别适用于需要遍历设备、故障、联锁、操作模式之间复杂关系的恢复规划。
- **仿真验证机制**：在物理执行前通过数字孪生评估动作可行性，避免直接对真实过程施加潜在危险动作，同时提供结构化反馈以辅助迭代修正。
- **通用性**：框架同时适用于离散（批处理）和连续（CSTR）过程，覆盖开环和闭环控制场景，表明方法具有一定泛化能力。
- **安全后备**：当LLM无法在有限时间窗口内找到有效方案时，自动移交安全后备，保障基本安全。
- **可解释性**：LLM能生成中间推理和动作理由，方便操作员审计。

## 8. 不足与局限

- **知识图谱构建成本高**：当前实现依赖于高度结构化的输入，工业实践中这些信息常以非结构化形式存在。自动构建知识图谱的方法超出研究范围，限制了实际部署的易用性。
- **未知故障处理能力缺失**：数字孪生只能模拟预先建模的故障，对于真正的未知故障（未建模的故障模式）无法仿真验证，恢复动作的安全性无法保证。
- **低恢复率故障暴露缺陷**：某些故障（如传感器故障、冷却故障）下GPT-4o-mini恢复率极低，表明重新提示循环无法克服根本推理错误，系统鲁棒性不足。
- **延迟不适用于硬实时系统**：即使轻量LLM（2-3秒/次决策）对于毫秒级安全关键任务不可接受。仅适用于秒级甚至更慢的过程动态。
- **可扩展性未验证**：仅测试了单个混合模块和单个CSTR，未涉及大型耦合工厂（多单元、多回路）。随着系统规模增大，知识图谱规模、推理复杂性、token消耗都会增长，框架的可扩展性未知。
- **实验充分性有限**：每故障仅10次重复，未提供统计显著性检验。未进行消融实验、敏感性分析或与经典FTC方法的定量对比。未测试开源模型或不同提示策略。基线实验仅确认无故障时正常运行，未与无LLM的自动化控制策略比较。
- **依赖云API**：实验中调用OpenAI API，存在延迟波动、成本不可控、数据隐私等问题。工业部署需本地边缘模型、安全隔离等额外措施。
- **优化目标仅为可行性而非最优性**：框架寻找第一个满足约束的修复动作，而非最优恢复路径，可能错过更优折中方案（如最小化产量损失）。
