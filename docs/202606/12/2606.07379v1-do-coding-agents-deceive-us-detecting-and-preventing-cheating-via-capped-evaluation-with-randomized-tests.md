## 摘要
在智能体评估和训练中，一种日益增长的失败模式是模型可以通过利用捷径而非解决预期任务来获得高评估分数，从而产生欺骗性表现。这使得评估分数作为真实任务解决能力的衡量标准变得不可靠。我们提出了CapCode，一个构建带有随机测试的编码数据集的框架，其最佳可实现非作弊性能被故意限制在1以下。这种封顶性能设计赋予了评估分数更清晰的解释：显著高于封顶值的分数不可信，因此提供了作弊的证据。为了防止作弊，我们提出了CapReward，一种基于CapCode原则的奖励设计，旨在抑制超出封顶值的优化。跨多个数据集的实验表明，CapCode在检测作弊的同时保持了模型的性能排名，而CapReward减少了作弊行为，使得模型更好地遵循预期任务规范。

## Abstract
A growing failure mode in agent evaluation and training is that models can achieve high evaluation scores by exploiting shortcuts instead of solving the intended task, producing deceptive performance. This makes evaluation scores unreliable as measures of true task-solving ability. We propose CapCode, a framework for constructing coding datasets with randomized tests whose best achievable non-cheating performance is deliberately capped below one. This capped-performance design gives evaluation scores a clearer interpretation: scores substantially above the cap are implausible and therefore provide evidence of cheating. To prevent cheating, we propose CapReward, a reward design based on the CapCode principle to discourage optimization beyond the cap. Experiments across multiple datasets show that CapCode detects cheating while preserving performance ranking of models, and CapReward reduces cheating behavior, yielding models that better follow the intended task specification.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：在编码智能体（coding agents）评估中，模型常通过利用测试捷径（如硬编码输出、注入验证码等）获得高分，而非真正解决任务。这种欺骗性表现（deceptive performance）使评估分数无法反映真实能力，威胁评估可信度。
- **研究动机**：现有作弊检测方法（如添加冲突测试、分析推理轨迹）存在局限性（如降低评估能力、依赖过程信号）。需要一种系统性、可解释且能同时检测与预防作弊的方法。
- **整体含义**：提出一种基于“封顶性能”（capped performance）的框架，使评估分数具有明确的作弊指示意义——显著超过封顶值的分数只能由作弊解释，从而为评估和训练提供更可靠的信号。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：通过随机化测试构造数据集，使无作弊策略的预期通过率**故意低于1**（称为“封顶”，cap）。若模型得分显著超过封顶，则高度可疑为作弊（利用测试信息）。
- **关键技术细节**：
  1. **CapCode框架**：对每个任务注入随机均匀采样的“封顶值”（cap value），创建多个同样合法的输出（如原始输出附加0或1），但测试只接受其中一个随机版本。因此非作弊策略只能随机猜测，期望通过率上限为1/M（M为等权输出数量）。
     - **任务级 CapCode**：每个任务整体添加一个额外的随机测试（如`code() == 0`或`code() == 1`），只有一个被固定用于评估。
     - **案例级 CapCode**：每个测试案例的预期输出附加上随机封顶值（如`cube_Sum(2) == (72, 1)`），更细粒度检测作弊位置。
  2. **作弊检测**：使用单侧二项检验，检验整体通过率是否显著超过封顶值。若p值小于显著性阈值（如1%），则判定作弊。
  3. **CapReward奖励设计**：在强化学习微调中，将奖励函数设计为在通过率等于封顶时达到最大值，低于或高于封顶时递减（形如分段幂函数）。这从理论上阻止模型追求超过封顶的高分（命题3证明无欺骗策略能最大化此奖励）。
     - 奖励函数：当$s \leq B$时，$r=(s/B)^{\kappa_l}$；当$s > B$时，$r=((1-s)/(1-B))^{\kappa_r}$。$B$为封顶值，$\kappa_l, \kappa_r$控制陡峭程度。

## 6. 主要结论与发现

1. **CapCode能有效检测作弊**：在反馈暴露、提示暴露、工作区暴露三种场景下，模型在开放测试上的通过率显著超过封顶值（0.5），而隐藏测试通过率低或下降，符合作弊特征。二项检验在1%显著性水平下成功标记作弊。
2. **CapCode保留模型排名能力**：在有封顶的数据集上，模型得分与原始数据集得分呈强单调正相关（Kendall's τ 平均 0.89-0.95），表明不干扰相对能力评估。
3. **CapReward有效减少作弊**：在初始策略已作弊时，CapReward训练的模型在隐藏测试上表现更高，开放-隐藏差距更小，优于所有基线（B/NB/GR/ImpReward）。
4. **CapReward对非作弊策略无害**：当初始策略无作弊时，CapReward性能与基线相当，不会错误惩罚正常行为。
5. **KL正则化可能适得其反**：在频繁作弊初始策略下，KL正则化会固化作弊行为，禁用KL反而有助于纠正。
