## 摘要
工具增强型语言模型代理通常通过它们是否选择了正确的工具、生成了有效的API参数以及完成了所请求的任务来评估。然而，一个代理可能选择了正确的工具，但仍然对错误的外部实体采取了行动。例如，请求“给Alex发邮件关于发布事宜”可能导致代理联系了错误的Alex、附加了错误的发布文档、在错误的线程中回复或更新了错误的客户账户。我们将这些错误称为实体绑定失败。本文研究了工具增强型代理中作为独立可靠性和安全问题的实体绑定失败。我们形式化了工具正确性与实体正确性之间的分离，引入了企业工作流中错误实体失败的分类法，并评估了实体感知的执行机制，包括实体解析前提条件、置信门控绑定、模糊情况下的澄清以及溯源跟踪。在跨60个任务、五个模型后端和六种工具使用方法的受控诊断评估中，所有方法均实现了0.0%的错误工具错误率，但面向动作的基线仍在24.0-26.0%的运行中产生了错误实体动作。实体感知方法在此设置中消除了错误实体动作和风险加权错误实体暴露，但通过在模糊情况下推迟处理降低了直接任务完成率。这些发现表明，安全使用工具不仅需要选择正确的工具，还需要在行动前可靠地将自然语言引用绑定到正确的现实世界实体。

## Abstract
Tool-augmented language-model agents are often evaluated by whether they select the correct tool, produce valid API arguments, and complete the requested task. However, an agent may choose the right tool and still act on the wrong external entity. For example, a request to "email Alex about the launch" may lead the agent to contact the wrong Alex, attach the wrong launch document, reply in the wrong thread, or update the wrong customer account. We call these errors entity binding failures. This paper studies entity binding failures as a distinct reliability and safety problem in tool-augmented agents. We formalize the separation between tool correctness and entity correctness, introduce a taxonomy of wrong-entity failures in enterprise workflows, and evaluate entity-aware execution mechanisms including entity-resolution preconditions, confidence-gated binding, clarification under ambiguity, and provenance tracking. In a controlled diagnostic evaluation across 60 tasks, five model backends, and six tool-use methods, all methods achieved 0.0 percent wrong-tool error, yet action-oriented baselines still produced wrong-entity actions in 24.0-26.0 percent of runs. Entity-aware methods eliminated wrong-entity actions and risk-weighted wrong-entity exposure in this setting, but reduced direct task completion by deferring under ambiguity. These findings show that safe tool use requires not only selecting the correct tool, but also reliably binding natural-language references to the correct real-world entity before action.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：工具增强型语言模型（LLM）代理在选取正确工具后，仍可能将自然语言引用绑定到错误的现实世界实体，导致严重安全与可靠性问题。例如，指令“给Alex发邮件关于发布事宜”中，代理可能选对了“发邮件”工具，却绑定到错误的Alex、错误的文档或错误的邮件线程。
- **背景与动机**：现有工具使用评估主要关注工具选择、API参数有效性和任务完成率，忽略了“工具正确但实体错误”这一独立失败模式。在企业工作流中，这种“正确工具、错误目标”的行为可能造成隐私泄露、运营故障、财务损失和声誉损害。尤其是在名称重复、时间模糊、跨系统引用等常见歧义场景下，代理若在绑定不确定时直接执行，风险极高。
- **整体含义**：安全工具使用不仅需要选择正确工具，更需要在执行前可靠地将自然语言引用解析并绑定到正确的现实世界实体。本文首次将实体绑定失败作为一个独立的可靠性和安全问题进行研究。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

- **核心思想**：在工具选择与工具执行之间插入一个**实体感知动作门（entity-aware action gate）**，强制要求在调用外部工具前先明确目标实体是否已充分解析。只有当实体绑定满足置信度和分离度条件时，才允许执行；否则应进行澄清或安全推迟。
- **关键技术细节**：
  - **实体解析前置条件（Entity-Resolution Preconditions）**：每个工具关联一组实体类型的前提条件（如发送邮件要求收件人实体是必需的，线程和附件是可选的）。只有所有必需前提条件都已绑定到具体实体ID时，才允许执行。
  - **候选实体检索**：对于每条实体提及，从环境中检索候选实体集合（通过精确匹配、语义相似度、时间过滤、所有权过滤等），保持召回率高，歧义交由下游门控处理。
  - **置信度门控绑定（Confidence-Gated Binding）**：对每个候选实体计算绑定得分s(m,e,S)，选择最高分实体作为预测绑定。只有满足两个条件时才视为已解析：最高分≥阈值τ，且最高分与次高分之差≥边际δ。这防止在弱证据或近歧义情况下贸然行动。
  - **实体感知动作门（Algorithm 1）**：算法流程为：选择候选工具→检索实体前提→提取必需提及→对每条提及检索候选集、打分、选最优、记录溯源→检查是否所有必需提及都已解析且前提满足→若满足则执行，否则返回澄清或推迟。
  - **溯源跟踪（Provenance Tracking）**：记录每次绑定的证据（实体ID、名称、邮箱、标题、时间戳等），支持审计和更好的澄清。
- **公式与算法**：论文给出了工具正确性、实体正确性、实体绑定失败、多实体绑定失败、风险加权绑定失败、歧义判定、解析判定等定义，以及实体感知动作门的伪代码。

## 3. 实验设计：数据集/场景、benchmark、对比方法

- **数据集/场景**：构建了一个受控诊断测试台（diagnostic testbed），包含60个任务，覆盖五个企业域：
  - 电子邮件（收件人、线程、附件）
  - 日历（事件、循环会议、实例）
  - 文档（版本、标题、所有者）
  - 客户记录（账户、子公司）
  - 问题跟踪（工单、bug、功能请求）
- **歧义类型**：包括无歧义、名称碰撞、文档版本歧义、时间歧义、账户碰撞、近重复实体、跨系统引用、真实歧义（无唯一目标可恢复）。
- **任务风险等级**：低（读/检索）、中（草稿/准备）、高（发送/分享/更新）、严重（删除/取消/关闭）。
- **对比方法**（共6种）：
  1. **Direct**：直接输出工具调用（动作导向基线）
  2. **Semantic filter**：按语义相关性过滤工具后输出
  3. **CMTF only**：基于因果最小工具过滤（Causal Minimal Tool Filtering），不含实体门控
  4. **Entity retrieval**：在工具调用生成时提供检索到的候选实体
  5. **Confidence gate**（实体感知）：仅当实体绑定充分支持时执行，否则推迟
  6. **Entity CMTF + provenance**（实体感知）：结合因果过滤、实体前提和溯源证据
- **模型后端**：5种模型（Amazon Nova 2 Lite、Amazon Nova Premier、Claude Opus、Claude Sonnet、Llama 3.3 70B Instruct）。
- **总运行数**：60任务 × 5模型 × 6方法 = 1800次运行。

- **实体绑定失败普遍存在**：所有方法工具错误率为0%，但动作导向基线仍有24.0–26.0%的运行出现实体绑定失败（正确工具、错误实体）。说明仅凭工具正确性评估远不足以保证安全。

- **最易失败的任务类型**：时间歧义任务和真实歧义任务（100%错误实体率），名称碰撞和跨系统引用也有20%错误率。无歧义任务均正确。

- **只评估单步工具调用**：未涉及多步推理、回溯或恢复场景，实际中早期实体绑定错误可能传播到后续步骤。
