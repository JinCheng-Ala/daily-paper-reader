## 摘要
多智能体人工智能系统越来越多地被用于自动化软件工程任务，包括需求分析、架构设计、测试生成和可追溯性链接。当这些智能体在共享软件工件上以顺序流水线方式运行时，上游智能体产生的错误和低置信度决策会传播到下游阶段，导致孤立的需求、矛盾的链接和合规性差距，这在安全关键领域构成重大风险。我们提出了一种基于信任的协调框架，其中共享知识图谱既充当集中式语义记忆，又作为协调表面，智能体通过校准的置信度分数评估并建立在彼此的贡献之上。我们的方法引入了一个两阶段的可追溯性链接预测流水线，结合了基于嵌入的检索与基于LLM的多标准分析，以及一种可追溯性种子机制，使得推导时和验证时的置信度得以比较，并通过置信度阈值门控、置信度发散检测和冲突解决来管理流水线交互的一致性协议。我们在一个汽车软件工程案例研究上进行评估，测量链接预测校准、协议有效性、阈值敏感性和可追溯性种子的影响。消融研究证实，置信度校准对于有效的水线协调至关重要。

## Abstract
Multi-agent AI systems are increasingly used to automate software engineering tasks including requirements analysis, architecture design, test generation, and traceability linking. When these agents operate as a sequential pipeline over shared software artifacts, errors and low-confidence decisions made by upstream agents propagate to downstream stages, producing orphaned requirements, contradictory links, and compliance gaps that pose significant risks in safety-critical domains. We propose a trust-aware coordination framework where a shared knowledge graph serves as both centralized semantic memory and a coordination surface through which agents assess and build upon each other's contributions using calibrated confidence scores. Our approach introduces a two-stage traceability link prediction pipeline combining embedding-based retrieval with LLM-based multi-criteria analysis, a traceability seeding mechanism that enables comparison between derivation-time and validation-time confidence, and a consistency protocol governing pipeline interactions through confidence threshold gating, confidence divergence detection, and conflict resolution. We evaluate on an automotive software engineering case study measuring link prediction calibration, protocol effectiveness, threshold sensitivity, and the impact of traceability seeding. Ablation studies confirm that confidence calibration is essential for effective pipeline coordination.

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- **信任感知协调**：将每个智能体的输出（可追溯性链接）携带校准的置信度分数，作为下游智能体决策的信号。通过比较推导阶段和验证阶段的置信度差异，检测不一致并生成冲突节点。

### 关键技术细节
1. **知识图谱架构**：构建有向标签属性图 \( G = (V, E, \phi, \psi) \)，节点类型包括需求、组件、测试、冲突；边类型包括REF_BY（派生）、IMP_BY（实现）、VER_BY（验证）、AFFECTS（影响）。每个边携带置信度 \( c(e) \in [0,1] \)、理由、创建智能体ID和状态。
2. **两阶段可追溯性链接预测**：
   - **阶段1：语义检索**：使用预训练编码器 \( M_{\text{enc}} \) 将每个工件嵌入，通过余弦相似度检索 Top-k 候选。
   - **阶段2：LLM多标准分析**：对每个候选对(\(r, c\))，由LLM评估四个标准（需求覆盖、实现特异性、术语对齐、无矛盾），输出链接决策 \( y_{rc} \)、置信度 \( c_{rc} \)、理由 \( j_{rc} \)。
3. **可追溯性种子机制**：需求智能体在派生软件需求时生成初始链接，记录推导置信度 \( c_{\text{seed}} \)。随后，可追溯性智能体独立验证这些链接，产生验证置信度 \( c_{\text{val}} \)，二者共存于同一边上。
4. **一致性协议**：包含三个机制：
   - **置信度阈值门控**：下游智能体不构建在低置信度（低于阈值）的链接上。
   - **置信度差异检测**：如果 \( |c_{\text{seed}} - c_{\text{val}}| > \delta \) 则触发冲突。
   - **冲突解决与物化**：将检测到的冲突（差异冲突、跨簇歧义冲突）作为图节点物化，带有类型、严重度、状态等属性。
5. **参数设置**：
   - 差异阈值 \( \delta = 0.5 \)（领域专家设定）。
   - 跨簇歧义冲突使用两个参数：置信度阈值 \( \tau \) 和簇间距离阈值 \( \epsilon \)，通过消融实验优化为 \( \tau=0.5, \epsilon=0.4 \)。
6. **工程分析**：通过Cypher查询进行影响分析、可追溯性缺口检测、测试覆盖率计算、中心性分析、需求聚类等。

### 公式与算法流程（文字说明）
- 智能体流水线：\( \Pi = A_{\text{req}} \to A_{\text{arch}} \to A_{\text{trace}} \to A_{\text{test}} \to A_{\text{analysis}} \)。
- 冲突检测条件：
  - 差异冲突：\( |c_{\text{seed}} - c_{\text{val}}| > \delta \)。
  - 跨簇歧义：对同一软件需求，其REF_BY链接到两个语义距离大于 \( \epsilon \) 的簇，且每个链接的置信度均 ≥ \( \tau \)。

1. **链接预测**：两阶段流水线在全局F1=0.769，显著优于TF-IDF (0.349) 和纯LLM (0.669)，主要得益于语义预筛选减少了上下文窗口饱和问题。
