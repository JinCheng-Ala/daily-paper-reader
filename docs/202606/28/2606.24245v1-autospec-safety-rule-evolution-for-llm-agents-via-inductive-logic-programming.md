## 摘要
大语言模型（LLM）智能体通过将语言模型与外部工具和环境集成，日益自动化地完成复杂任务。然而，它们的自主性带来了显著的安全风险：智能体可能执行破坏性命令、泄露敏感数据或违反领域约束。现有安全方法面临根本性权衡：手工制定的规则可解释但脆弱，过于保守的规则会阻止安全操作（高假阳性），而宽松的规则则遗漏不安全行为（高假阴性）。神经分类器缺乏安全关键部署所需的可解释性。我们提出AutoSpec，一个通过归纳逻辑编程（ILP）指导的反例引导归纳合成（CEGIS）框架，能够从用户的安全/不安全标注中自动演化部署的专家设计安全规则。从专家规则和标注迹流开始，AutoSpec迭代评估规则，挖掘假阳性和假阴性反例，使用ILP学习哪些谓词能够区分它们，生成候选规则编辑，并验证候选者以选择最佳修订。关键洞察在于，ILP能高效识别在假阴性中频繁出现但在假阳性中很少出现（或反之）的谓词，从而大幅剪枝规则编辑的指数级搜索空间。该过程持续进行直至收敛，产生平衡精确率和召回率的可解释规则。我们在涵盖代码执行和具身智能体领域的291条执行迹上评估了AutoSpec。AutoSpec在这两个领域将规则F1值分别提升至0.98和0.93，在保持高召回率的同时实现了高达94%的假阳性减少，并在4-5次迭代内收敛。ILP引导的方法相比启发式CEGIS实现了高达4.8倍的F1值提升。学习到的规则可读、可审计，并能泛化到未见场景。

## Abstract
Large language model (LLM) agents increasingly automate complex tasks by integrating language models with external tools and environments. However, their autonomy poses significant safety risks: agents may execute destructive commands, leak sensitive data, or violate domain constraints. Existing safety approaches face a fundamental tradeoff: hand-crafted rules are interpretable but brittle, with overly conservative rules blocking safe operations (high false positives) while permissive rules miss unsafe behaviors (high false negatives). Neural classifiers lack the interpretability required for safety-critical deployments.   We present AutoSpec, a framework that automatically evolves deployed expert-designed safety rules from user safe/unsafe annotations through counterexample-guided inductive synthesis (CEGIS) guided by inductive logic programming (ILP). Starting from the expert rules and a stream of annotated traces, AutoSpec iteratively evaluates rules, mines false-positive and false-negative counterexamples, uses ILP to learn which predicates discriminate them, generates candidate rule edits, and verifies candidates to select the best revision. The key insight is that ILP efficiently identifies predicates that appear frequently in false negatives but rarely in false positives (or vice versa), dramatically pruning the exponential search space of rule edits. This continues until convergence, producing interpretable rules that balance precision and recall.   We evaluate AutoSpec on 291 execution traces spanning code execution and embodied agent domains. AutoSpec raises rule F1 to 0.98 and 0.93 across the two domains, achieving up to 94% false positive reduction while maintaining high recall, and converges within 4-5 iterations. The ILP-guided approach achieves up to 4.8x higher F1 than heuristic CEGIS. The learned rules are human-readable, auditable, and generalize to unseen scenarios.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**：大语言模型（LLM）智能体在执行任务时具有高度自主性，但可能执行破坏性命令、泄露敏感数据或违反领域约束，引发严重安全风险。现有安全方法存在根本性权衡：
  - 手工制定的规则（如 AgentSpec）：可解释、可审计，但静态且脆弱——过于保守导致高假阳性（FP），过于宽松导致高假阴性（FN），难以随环境演进。
  - 神经分类器（如 Llama Guard）：更具适应性，但缺乏可解释性，无法满足安全关键部署对可审计性的要求。
- **挑战**：随着模型更新、工具集成、提示词修订，安全行为的边界持续变化，固定规则集逐渐失效。同时，智能体运行时积累的用户安全/不安全标注（如批准或拒绝工具调用）恰好为规则演化提供了宝贵信号。
- **目标**：从初始专家规则出发，利用标注迹自动演化出更精确、可解释、可审计的安全规则，将规则维护从手动、一次性任务转变为连续的、标注驱动的演化问题。

## 2. 方法论：核心思想、关键技术细节与算法流程

- **核心思想**：将安全规则演化转化为反例引导的归纳合成（CEGIS）问题，用户标注作为反例（FP/FN），通过归纳逻辑编程（ILP）高效学习区分反例的谓词，指导规则编辑搜索，从而在大幅剪枝搜索空间的同时保持规则可解释性。
- **关键技术细节**：
  - **谓词库P**：每个谓词是一个布尔函数，判断事件是否具有特定属性（如`destructive_os_inst`、`near_danger_zone`）。规则由这些谓词组合而成（合取、例外、析取分支）。
  - **规则评估与反例挖掘**：对每条迹，若任何规则在任何事件上触发则标记为不安全。比较预测与真实标签得到FP（假阳性，安全迹被误判为不安全）和FN（假阴性，不安全迹被漏判）。
  - **ILP谓词学习**：使用ILASP求解器，以FN为正例、FP为反例，学习一个由最少谓词组成的逻辑假设（析取范式），能够区分两类反例。输出的每个谓词附带方向：若帮助覆盖FN则用于放宽（AddDisjunct/Relax），若帮助排除FP则用于收紧（AddConjunct/AddException）。
  - **候选生成**：基于ILP建议，通过三类编辑策略生成候选规则集：
    1. 单步编辑：直接应用一个建议。
    2. 混合修复链：先收紧以消除FP，再放宽以覆盖FN，避免修复FN引入新FP。
    3. 失败驱动分支：对残余FN使用精度约束的贪婪集合覆盖，添加高精度析取分支。
  - **候选选择**：对每个候选在全部标注集上评估，按字典序优化：F1 > 精度 > 未解决反例数 > 规则数，保留表现最佳的候选。
  - **收敛机制**：通过软多样性压力（对已出现谓词降权）避免循环，当无候选能提升分数时停止。
- **算法流程（Algorithm 1）**：
  - 输入：初始规则R0、标注迹D_L、谓词库P、最大迭代K、目标阈值θ。
  - 循环 i=1 to K：
    1. 若S(R) ≥ θ，返回R。
    2. 从R和D_L中挖掘反例CE（FP和FN）；若CE为空，返回R。
    3. 使用ILP从CE中学到建议（discriminating predicates）。
    4. 基于建议生成候选规则。
    5. 在D_L上评估候选，选择最佳best。
    6. 若S(best) > S(R)，更新R=best；否则返回R。

## 3. 实验设计：数据集、场景与对比方法

## 6. 主要结论与发现

- **有效性**：AutoSpec在代码执行域和具身智能域分别达到F1 0.980和0.933，相比初始专家规则（B1）提升40.4%和89.2%。假阳性减少达94%，同时保持高召回率。
- **ILP效率优势**：ILP引导的CEGIS相比无ILP的启发式版本，F1提升4.8倍（代码域0.980 vs 0.203），且收敛更快、泛化更好。
- **收敛速度**：代码域4次迭代内达到F1≥0.95，具身域5次迭代收敛，表明ILP能快速定位关键谓词。
- **泛化能力**：从初始70%任务学到的规则在入站30%任务上性能下降极小（代码域1.6%、具身域4.7%），说明规则捕获了通用不安全模式而非记忆。
- **可解释性与信任**：用户研究显示，AutoSpec规则在可解释性（4.97 vs 3.45）、可操作性（4.83 vs 2.73）、信任度（4.80 vs 2.77）上均显著优于LLM分类器（p<0.01）。从业者高度认可其可审计性和确定性。
- **消融重要结论**：ILP引导是性能提升的最关键组件（去除后F1从0.98降至0.20）；多步编辑和失败驱动分支显著提升效率（减少迭代次数和候选数）。

## 7. 优点

- **创新性**：首次将安全规则维护形式化为ILP引导的CEGIS问题，利用反例标注自动演化可解释规则，而非从头合成或丢弃专家知识。
- **高效剪枝**：ILP能够从指数级谓词组合空间中快速识别少数判别性谓词，避免了枚举暴力搜索，使迭代收敛快（4-5轮）。
- **实用性**：输出规则保持与初始规则相同的可读、可审计形式，零推理开销，可直接部署到现有防御栈中；且与神经方法互补，提供确定性保障。
- **稳健性与泛化**：通过多样性压力和单调进行保证，规则在未见任务上性能下降很小，避免了对训练集的过拟合。
- **用户验证**：通过从业者主观评估确认了符号规则相比黑盒分类器在可操作性、信任度上的巨大优势，补充了客观指标。

## 8. 不足与局限

- **领域覆盖有限**：仅验证了代码执行和具身智能体两个领域，未涉及Web、数据库、多智能体协作等场景。结论的泛化性待进一步验证。
- **谓词库依赖**：系统依赖专家预定义的谓词库（16-24个谓词/域），谓词的质量和覆盖范围直接影响演化上限。不完整的谓词库可能导致遗漏重要安全模式。
- **时序与状态依赖**：当前规则仅在单个事件上评估谓词，无法直接表达跨事件的时序约束（如“读取凭证后调用外部API”）。虽然作者指出可通过谓词内部编码，但原始表达能力受限。
- **标注质量假设**：论文假设标注迹的标签完全正确。如果用户标注有偏差，演化出的规则可能继承这些偏差。实际中可能存在标注不一致或恶意标注。
- **局部最优性**：采用贪婪逐轮更新，不保证全局最优。结果依赖于初始规则和ILP超时设置，可能在某些复杂场景下收敛到次优解。
- **计算开销**：ILP求解在最坏情况下有120秒超时，虽然不是每次都需要，但多次迭代可能累积显著时间（代码域20秒）。对于需要实时更新的场景可能不够快。
- **未与其他可解释方法比较**：仅对比了LLM分类器，未与决策树、规则学习器（如RIPPER）或可解释AI方法直接比较。
